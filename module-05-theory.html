5.1 · Theory

RAG Evaluation: The Three Quality Pillars

A RAG pipeline has two distinct stages — retrieval and generation — and each can fail independently. You can't evaluate RAG with a single quality score. Instead, evaluation splits into three pillars, each targeting a different point in the pipeline. This decomposition is what enables root-cause diagnosis: when something goes wrong, you know where it broke.

The three pillars

Pillar 1 — Retrieval
Did we fetch the right documents?
Scored by RetrievalRelevance. Checks whether the documents pulled from your vector store actually relate to the user's question. If retrieval fails, generation has no chance — the answer will be fabricated or generic regardless of how good the LLM is.
Pillar 2 — Groundedness
Is the answer grounded in context?
Scored by RetrievalGroundedness. Checks whether every claim in the response can be traced back to the retrieved chunks. This is the hallucination detector — a high score means the model is "reading" the context, not making things up.
Pillar 3 — End-to-end
Does the answer address the question?
Scored by Correctness + RelevanceToQuery. Operates on the final output, treating the pipeline as a black box. This is what the user experiences — factual accuracy and topic relevance of the response.

Key insight

Why three pillars, not one?

A single quality score tells you something is wrong but not where. The three pillars let you pinpoint root cause. The combination of Correctness (end-to-end) and Groundedness (retriever-generator boundary) forms a diagnostic matrix that points directly at the fix.

Correctness: LOW Groundedness: HIGH
Retrieval miss
The LLM faithfully used the retrieved context — but the retriever fetched the wrong documents. The model answered from bad context, not from hallucination.
Fix: Improve retrieval — re-chunk documents, update embeddings, add missing docs to the index.
Correctness: LOW Groundedness: LOW
Hallucination
The LLM ignored the retrieved context and generated claims that can't be traced back to any document. The context may have been fine — the model just didn't use it.
Fix: Improve generation — adjust the system prompt for stricter grounding, lower temperature, or switch models.
Practical workflow

Always fix retrieval first. If the retriever scores are consistently low, improving generation won't help — the LLM simply doesn't have the right context. Fix the chunking or embedding strategy, then re-evaluate. Only tune the generation prompt once retrieval scores are healthy.

Running all three pillars in one call

In MLflow, all three pillars are just scorers passed to a single evaluate() call. The trace-based scorers (RetrievalGroundedness, RetrievalRelevance) automatically extract the retrieval and generation spans from your traced RAG pipeline — you don't manually pass retrieved documents.

from mlflow.genai.scorers import ( Correctness, RelevanceToQuery, RetrievalGroundedness, RetrievalRelevance ) import mlflow # All three pillars in one evaluate() call results = mlflow.genai.evaluate( data=eval_dataset, predict_fn=my_rag_agent, scorers=[ RetrievalRelevance(), # Pillar 1: retrieval quality RetrievalGroundedness(), # Pillar 2: grounding fidelity Correctness(), # Pillar 3a: factual accuracy RelevanceToQuery(), # Pillar 3b: topic relevance ] ) # Diagnose: find rows where retrieval failed df = results.eval_table retrieval_misses = df[ (df["correctness"] <= 2) & (df["retrieval_groundedness"] >= 4) ] print(f"Retrieval misses: {len(retrieval_misses)} rows")
Trace-based scorers like RetrievalGroundedness require your agent to emit proper traces with RETRIEVER and CHAT_MODEL spans. Without mlflow.openai.autolog() or @mlflow.trace, these scorers silently return null — no error, just missing data.

5.3 · Theory

Agent Tracing Deep Dive — Reading Spans

Tracing is how you see inside your agent. Instead of treating the pipeline as a black box, MLflow tracing breaks every execution into spans — individual units of work that capture what happened, what went in, what came out, and how long it took. Understanding span structure is prerequisite for writing trace-based scorers and diagnosing agent behaviour.

What is a span?

A span represents a single operation within your agent's execution. When your RAG agent runs, it doesn't do one thing — it retrieves documents, calls an LLM, maybe invokes a tool, then formats a response. Each of those steps becomes a span in the trace. Spans are nested: a parent CHAIN span wraps the entire agent call, and child spans sit inside it for each sub-operation. Every span captures four things:

Captured
Inputs
What was passed into this operation — the query for a retriever, the messages array for an LLM call, the arguments for a tool invocation.
Captured
Outputs
What the operation returned — retrieved documents, the LLM response, the tool result. This is what downstream scorers read.
Captured
Timing
Start and end timestamps in nanoseconds. Enables latency analysis per span — find which operation is the bottleneck.
Captured
Attributes
Metadata like model name, token count, endpoint URL, or custom key-value pairs you attach via the tracing API.

Span types

MLflow defines specific span types so scorers and the UI know what each span represents. Click each type below to see what it captures and when it appears.

CHAIN

Outermost wrapper

The root span representing the full agent execution from input to output. If your agent calls sub-agents, each gets its own nested CHAIN span.

InputsThe original user query or function arguments
OutputsThe final agent response (return value of your function)
TimingTotal wall-clock duration of the entire agent call
ChildrenAll other spans nest inside this — RETRIEVER, CHAT_MODEL, TOOL, etc.
Created by@mlflow.trace decorator on your agent function

CHAT_MODEL

LLM call · Chat format

An LLM call using the chat completions format. Captures the full messages array as input and the model's response as output. This is what RetrievalGroundedness reads from.

InputsThe messages array (system, user, assistant turns)
OutputsThe model's response message with content and role
AttributesModel name, token usage (prompt + completion), temperature
Created bymlflow.openai.autolog() or OpenAI client instrumentation
Used byRetrievalGroundedness, Correctness (reads response)

RETRIEVER

Document retrieval

A document retrieval operation — vector search, keyword search, or hybrid. Captures the query and the list of retrieved documents. This is the primary data source for RetrievalRelevance.

InputsThe search query passed to the retriever
OutputsList of retrieved documents (content + metadata)
AttributesIndex name, top-k, similarity threshold
Created byLangChain autolog, or manually via @mlflow.trace(span_type="RETRIEVER")
Used byRetrievalRelevance, RetrievalGroundedness (reads docs)

TOOL

External tool call

An external tool invocation — API call, database query, calculator, code execution. Captures the tool name, arguments, and result. Key for evaluating agent efficiency.

InputsTool name and arguments (the function call parameters)
OutputsThe tool's return value or result
TimingExternal call latency — often the largest contributor to total time
Created byLangChain/LangGraph autolog, or manually decorated tool functions
Used byToolCallEfficiency, custom trace-based scorers

EMBEDDING

Embedding model call

A call to an embedding model to convert text into a vector. Typically a child of a RETRIEVER span — the query gets embedded before the similarity search.

InputsThe text to embed (query or document chunk)
OutputsThe resulting vector (float array)
AttributesModel name, vector dimension, batch size
Created byEmbedding client autolog or manual decoration

LLM

Raw text completion

A raw text completion call (non-chat format). Less common in modern agents but still used for some backends that use the completions API rather than chat completions.

InputsThe prompt string (single text, not messages array)
OutputsThe completion text
AttributesModel name, token usage, generation parameters
NoteMost modern agents use CHAT_MODEL instead. LLM is for legacy or non-chat endpoints.

How spans nest inside a trace

A trace is a tree of spans. The outermost CHAIN span is the root. Child spans nest inside it in the order they execute. This nesting is what lets you see the complete execution structure — which operations happened, in what sequence, and how long each took.

API

The key API: trace.search_spans()

This is the method you'll use most when writing trace-based scorers. It lets you query a trace for specific span types and extract their data programmatically. This is what makes trace-based evaluation powerful — you can inspect intermediate steps, not just the final output.

from mlflow.entities import SpanType # Inside a trace-based scorer: @scorer def retriever_latency_check(trace): # Find all RETRIEVER spans in this trace retriever_spans = trace.search_spans( span_type=SpanType.RETRIEVER ) if not retriever_spans: return Feedback( value="no_retriever", rationale="No retriever span found in trace" ) # Check latency of the first retriever span span = retriever_spans[0] latency_ms = (span.end_time_ns - span.start_time_ns) / 1e6 return Feedback( value=latency_ms < 500, # Pass if under 500ms rationale=f"Retriever latency: {latency_ms:.0f}ms" )
# Other common search_spans patterns: # Count total tool calls tool_spans = trace.search_spans(span_type=SpanType.TOOL) num_tools = len(tool_spans) # Get token count from all LLM calls llm_spans = trace.search_spans(span_type=SpanType.CHAT_MODEL) total_tokens = sum( s.attributes.get("usage.total_tokens", 0) for s in llm_spans ) # Get retrieved documents for custom relevance check ret_spans = trace.search_spans(span_type=SpanType.RETRIEVER) docs = ret_spans[0].outputs if ret_spans else []

How tracing gets instrumented

There are two approaches depending on your framework. They compose — autolog captures the LLM calls inside your manually-traced agent function, giving you a complete picture.

Zero-code
Autolog
mlflow.openai.autolog() automatically instruments all OpenAI-compatible client calls. Each chat.completions.create() produces a CHAT_MODEL span. LangChain/LangGraph have their own autolog for the entire chain.
Best for: Quick setup, standard frameworks, when you want tracing without code changes.
Manual
@mlflow.trace decoration
Decorate your functions to create CHAIN spans. MLflow captures the function arguments as inputs and the return value as output. You control exactly what gets traced and which spans are created.
Best for: Custom agent frameworks, fine-grained control, when you need specific span types or custom attributes.
# Autolog: zero-code instrumentation mlflow.openai.autolog() # All OpenAI calls become CHAT_MODEL spans # Manual: explicit control @mlflow.trace def my_rag_agent(question: str) -> str: # This creates a CHAIN span wrapping everything docs = retrieve_docs(question) # If using autolog, RETRIEVER span auto-created response = client.chat.completions.create( model="databricks-claude-sonnet-4", messages=[ {"role": "system", "content": f"Context: {docs}"}, {"role": "user", "content": question} ] ) # CHAT_MODEL span auto-created by autolog return response.choices[0].message.content

Reading traces in the Databricks UI

The MLflow trace UI in Databricks shows each trace as a timeline view. The CHAIN span stretches across the full width representing total execution time. Child spans appear as nested bars underneath, proportional to their duration.

This timeline is useful for spotting performance bottlenecks — if the RETRIEVER span takes 80% of total time, you know where to optimise. It also reveals unexpected behaviour: an agent that makes three TOOL calls where one should suffice, or a CHAT_MODEL span that received truncated context because the retriever returned too much text.

Critical implementation detail

Remember from Module 1: ENABLE_MLFLOW_TRACING=True (or calling mlflow.deploy()) is critical. Without it, Inference Tables capture only request/response pairs — not the full trace JSON. Trace-based scorers like RetrievalGroundedness need the span tree to find RETRIEVER outputs. No trace means these scorers silently return null.

from mlflow.genai.scorers import ToolCallEfficiency from mlflow.genai import scorer from mlflow.entities import SpanType, Feedback @scorer def tool_call_count(trace): """Flag agents making too many tool calls for simple queries.""" tool_spans = trace.search_spans(span_type=SpanType.TOOL) count = len(tool_spans) return Feedback( value=count <= 3, rationale=f"Agent made {count} tool calls" ) # Combine built-in + custom in one evaluate() call results = mlflow.genai.evaluate( data=eval_dataset, predict_fn=my_agent_v2, scorers=[ ToolCallEfficiency(), # Built-in agent judge tool_call_count, # Custom trace-based scorer Correctness(), # Still check end-to-end quality ], model_id="models:/my-agent/2" # Tag as v2 for comparison )
