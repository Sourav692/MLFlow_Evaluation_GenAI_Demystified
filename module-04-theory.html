4.1 · Theory

Scorer Taxonomy: Judges vs. Code-Based vs. Guidelines

In MLflow 3, a Scorer is the universal interface for evaluating LLM outputs. Every scorer receives some combination of inputs, outputs, expectations, and the execution trace — then returns a quality signal. But scorers split into fundamentally different types depending on how they produce that signal. Understanding this taxonomy is essential because it determines what you can evaluate, where the scorer can run, and what it costs.

At the top sits the Scorer interface — a common contract that evaluate() understands. Beneath it, two branches diverge based on how they compute their scores. LLM Judges use another language model to assess quality semantically — they excel at subjective, nuanced judgments like "Is this answer correct?" or "Is the response grounded in the retrieved context?" Code-Based Scorers are plain Python functions decorated with @scorer that execute deterministic logic — regex checks, length calculations, latency thresholds, JSON validation. No LLM call, no cost per evaluation, no variance. Guidelines are a specialised flavour of LLM judge optimised for binary pass/fail compliance checks against a flat list of rules.

Interactive · Scorer taxonomy hierarchy Click nodes for details
Code-based @scorer functions cannot be registered for production monitoring — only LLM judges (built-in, make_judge(), and Guidelines) support .register().start(). Your latency check and JSON format check run in dev and CI, but cannot run as continuous production monitors. Design your scorer suite with this boundary in mind from the start.
Decision

When to use each scorer type

The decision rule is straightforward: if you can express it in Python without an LLM, use a code scorer. If the evaluation requires reading comprehension or semantic understanding, use an LLM judge. If you have a flat list of compliance rules that demand binary pass/fail verdicts, use Guidelines.

Dimension LLM judges Code-based (@scorer) Guidelines
What it checks Semantic quality — correctness, relevance, groundedness, tone Deterministic properties — length, format, latency, regex matches Compliance rules — policy adherence, format requirements
Cost per eval Token cost LLM call per row Zero Python execution only Token cost LLM call per row
Variance Non-deterministic — runs can yield slightly different scores Zero variance — identical results every time Low variance — binary pass/fail reduces noise
Production Supported Not supported Supported
Return type Feedback with rationale Primitive or Feedback Feedback with per-rule verdict
Best for Quality assessment, hallucination detection, factual accuracy SLA checks, format validation, business rule enforcement Policy compliance, style rules, safety guardrails
Cost architecture principle

Use code scorers as the first filter — they run at 100% coverage for zero cost. Only escalate to LLM judges for rows that pass the deterministic checks. In production, sample LLM judges at 10–50% of traffic. This tiered approach prevents the cost multiplication trap: 5 judges × 10,000 traces/day = 50,000 LLM calls just for evaluation.

Returns

Return types: primitives vs. Feedback objects

Scorers can return values in two forms. Understanding the difference matters because it determines how much debugging information you get when investigating low-scoring rows.

Return type 1
Primitive
Returns int, bool, str, or float directly. Just the score, no metadata. Fast to write, sufficient for simple metrics.
Return type 2
Feedback object
Returns Feedback(value, rationale, source). Bundles score with explanation and AssessmentSource recording who scored it.
# Primitive return — just the number @scorer def word_count(outputs: str) -> int: return len(outputs.split()) # Primitive return — boolean pass/fail @scorer def is_ascii(outputs: str) -> bool: return outputs.isascii()

The simplest approach. You get the value in the evaluation results table, but when a row scores poorly, there's no context about why. Suitable for quick development iteration or metrics where the value speaks for itself.

from mlflow.entities import Feedback, AssessmentSource @scorer def has_citation(outputs: str) -> Feedback: found = "[docs]" in outputs or "[ref]" in outputs return Feedback( value="yes" if found else "no", rationale="Citation tag found" if found else "No citation tag — agent should cite sources", source=AssessmentSource( source_type="CODE", source_id="citation_checker_v1" ) )

The Feedback object powers the MLflow UI's detailed view. When you see a row scored "no" for citations, you immediately see why and who scored it. The source_type field — CODE, LLM_JUDGE, or HUMAN — lets the UI group and compare automated scores against human feedback. This is the format all built-in judges use internally.

Feedback object anatomy

The Feedback object is the universal output format. Every built-in judge, every make_judge() custom judge, and every human label through the Review App or log_feedback() produces the same structure. This uniformity is what enables the MLflow UI to show automated scores and human labels side by side.

Field 1 · Required
value
The score itself. Can be an integer (1–5 rubric), boolean (pass/fail), string ("yes"/"no"), or float. Built-in judges use "yes"/"no" for binary and integers for rubric-based assessments.
Field 2 · Recommended
rationale
A written explanation of why this score was assigned. For LLM judges: chain-of-thought reasoning. For code scorers: you write it — e.g., "Latency was 3200ms, within 5000ms SLA".
Field 3 · Recommended
source
An AssessmentSource with source_type ("CODE", "LLM_JUDGE", "HUMAN") and source_id (e.g. "latency_v1"). Powers filtering and grouping in the UI.
Unified schema advantage

Because human feedback (Review App, log_feedback()), automated LLM judge scores, and code scorer results all produce the same Feedback shape, you can directly compare them. "The Correctness judge scored 4/5 but the human reviewer scored 2/5" — that gap is exactly what judge alignment (Module 7) measures and calibrates.

Lab: Built-in Judges — Run All on Tutorial Dataset

MLflow 3 ships with five built-in judges that cover the most common quality dimensions. Each is pre-configured with a prompt template, rubric, and scoring logic — you instantiate them and pass them to evaluate(). This lab applies all five to the dataset you built in Module 2 and teaches you how to interpret the results.

Built-in judges
Correctness
RelevanceToQuery
RetrievalGroundedness
Safety
Guidelines

Correctness

LLM Judge · Rubric 1–5

Evaluates whether the agent's response is factually accurate compared to the expected facts in your dataset. This is the primary quality metric for most Q&A and RAG applications.

Requiresexpectations.expected_facts in your dataset
ReturnsScore 1–5 with rationale explaining which facts were present, missing, or incorrect
Metric keycorrectness/mean, correctness/variance
ProductionSupported via .register().start()
Best forQ&A accuracy, factual verification, regression testing against known answers

RelevanceToQuery

LLM Judge · Rubric 1–5

Assesses whether the response actually addresses the user's question. A response can be factually correct but completely irrelevant — this judge catches that failure mode.

Requiresinputs (the original question) — no expectations needed
ReturnsScore 1–5 with rationale about how well the response addresses the query
Metric keyrelevance_to_query/mean
ProductionSupported via .register().start()
Best forDetecting off-topic responses, tangential answers, prompt injection bypasses

RetrievalGroundedness

LLM Judge · Rubric 1–5

For RAG pipelines: checks whether the agent's response is grounded in the retrieved context or whether it hallucinated information not present in the documents. This is the hallucination detector.

RequiresTrace with a RETRIEVER span containing retrieved documents
ReturnsScore 1–5 with rationale identifying which claims are grounded vs. fabricated
Metric keyretrieval_groundedness/mean
ProductionSupported via .register().start()
Best forHallucination detection, RAG faithfulness, preventing confabulation

Safety

LLM Judge · Binary yes/no

Checks whether the response contains harmful, toxic, or policy-violating content. Returns a binary "safe" or "unsafe" verdict. Typically run at 100% coverage with zero tolerance for failures.

Requiresoutputs only — no expectations needed
Returns"yes" (safe) or "no" (unsafe) with rationale
Metric keysafety/ratio_pass — should always be 1.0
ProductionSupported — run at sample_rate=1.0
Best forPolicy enforcement, harmful content detection, compliance requirements

Guidelines

LLM Judge · Pass/fail per rule

Evaluates the response against a flat list of compliance rules you define. Each rule gets an independent pass/fail verdict. The fastest path to enforcing tone, format, and policy constraints.

RequiresA list of guideline strings at instantiation time
ReturnsPer-rule "yes"/"no" with rationale per rule
Metric keyguidelines/ratio_pass (across all rules)
ProductionSupported via .register().start()
Best forStyle rules, language requirements, topic restrictions, policy compliance

The lab runs all five judges in a single evaluate() call on your UC-backed dataset from Module 2. You'll learn to interpret the results table — per-row scores, aggregate means, and the rationale for each judge — then identify the worst-scoring rows and diagnose root causes from the trace.

from mlflow.genai.scorers import ( Correctness, RelevanceToQuery, RetrievalGroundedness, Safety, Guidelines ) import mlflow # Load dataset from Module 2 dataset = mlflow.genai.datasets.get_dataset("main.genai_eval.tutorial_eval_v1") # Run ALL built-in judges in one call results = mlflow.genai.evaluate( data=dataset, predict_fn=my_agent, scorers=[ Correctness(), RelevanceToQuery(), RetrievalGroundedness(), Safety(), Guidelines(guidelines=[ "Response must be in English", "Must not recommend deleting production tables", ]) ], model_id="models:/my-agent/1" ) # Inspect aggregate metrics print(results.metrics) # → {'correctness/mean': 4.1, 'safety/ratio_pass': 1.0, ...} # Compare two prompt versions in MLflow Experiments view # Run again with a new prompt → different run_id # Open Experiments UI → select both runs → Compare view
Root-cause diagnosis technique

When a row scores correctness=2 but retrieval_groundedness=5, the retriever fetched the right documents but the LLM generated an inaccurate answer — a generation failure, fix the system prompt. When both are low, it's a retrieval failure — fix the vector search index or embeddings. The combination of judges is what enables root cause analysis.

4.3 · Hands-On Lab · New

Lab: Code-Based Scorers with @scorer Decorator

LLM judges are powerful but expensive and non-deterministic. For business rules with unambiguous right/wrong answers — response format, latency SLAs, citation presence, character limits — a Python function is faster, cheaper, and more reliable. The @scorer decorator turns any function into a scorer that plugs directly into evaluate().

Patterns

Three code-based scorer patterns

Code scorers access different parts of the evaluation context depending on what they need to check. The function parameter names determine what MLflow injects — this is the same key-matching contract as predict_fn.

Pattern 1 — Output-only, primitive return
Checks properties of the agent's text output. Declare outputs: str — MLflow injects the agent's response. Return int, bool, str, or float.
outputs → int
Pattern 2 — Output-only, Feedback return
Same input, but returns a Feedback object with value, rationale, and AssessmentSource(source_type="CODE"). Provides debugging context.
outputs → Feedback
Pattern 3 — Trace-based
Accesses the full execution trace — latency of specific spans, number of tool calls, token counts. Declare trace param, use trace.search_spans(span_type=...).
trace → Feedback
from mlflow.genai import scorer from mlflow.entities import Feedback, AssessmentSource, SpanType # ── Pattern 1: Primitive return ── @scorer def word_count(outputs: str) -> int: return len(outputs.split()) # ── Pattern 2: Feedback with rationale ── @scorer def has_code_block(outputs: str) -> Feedback: passed = "```" in outputs return Feedback( value="yes" if passed else "no", rationale="Response includes a code example" if passed else "No code block found — technical answers should include examples", source=AssessmentSource(source_type="CODE", source_id="code_checker_v1") ) # ── Pattern 3: Trace-based latency check ── @scorer def latency_ok(trace) -> Feedback: spans = trace.search_spans(span_type=SpanType.CHAT_MODEL) if not spans: return Feedback(value=False, rationale="No CHAT_MODEL span found") span = spans[0] ms = (span.end_time_ns - span.start_time_ns) / 1e6 return Feedback( value=ms <= 5000, rationale=f"LLM call took {ms:.0f}ms (SLA: 5000ms)", source=AssessmentSource(source_type="CODE", source_id="latency_sla_v1") )
Interactive · Available function parameters Hover for details
The function parameter names must exactly match: inputs, outputs, expectations, trace. If you name a parameter response instead of outputs, MLflow won't inject it and you'll get a TypeError. This is the same contract as predict_fn inputs matching dataset keys.
4.4 · Hands-On Lab

Lab: Custom LLM Judge with make_judge() on Databricks

Built-in judges cover the common dimensions, but your domain has specific quality criteria that generic judges can't assess. A "Databricks Technical Accuracy" judge needs to know that Z-ordering is about multi-dimensional clustering, not alphabetical sorting. make_judge() lets you define exactly what quality means for your domain.

Anatomy

The four components of a custom judge

Component 1
name
Unique identifier for this judge. Becomes the metric key prefix in results — e.g., "databricks_accuracy" produces databricks_accuracy/mean.
required
Component 2
instructions
The judge prompt template. Use {{ inputs }}, {{ outputs }}, {{ expectations }} as template variables. Must include a clear rubric with concrete level descriptions.
critical
Component 3
model
The LLM endpoint acting as judge. On Databricks: "databricks:/databricks-claude-sonnet-4". Frontier models for pre-deploy gates, smaller models for high-volume monitoring.
configurable
Component 4
examples (optional)
Few-shot anchors with human-rated examples. These calibrate the judge to match human scoring standards. Critical for alignment — without anchors, the judge defines its own interpretation of each score level.
alignment

Writing effective rubrics

The rubric is the single most important component. A vague instruction like "score 1–5 based on quality" gives the judge no calibration — scores will be arbitrary and unreproducible. Every level must describe specific, observable criteria.

Example rubric · Databricks technical accuracy
5
Precise, complete, includes caveats. All technical claims are correct. Mentions relevant limitations or edge cases. Uses proper terminology (e.g., "Z-ordering optimises data layout for multi-dimensional clustering using Hilbert curves").
4
Mostly correct, minor omissions. Core concept is right. May miss a secondary detail (e.g., describes Z-ordering correctly but doesn't mention it only works on Delta tables).
3
Partially correct, key concept present but incomplete. Gets the general idea right but misses essential details or oversimplifies in a misleading way.
2
Significant inaccuracy. Contains a factual error that would mislead a user — e.g., "VACUUM deletes data from the current table version" (it only removes unreferenced files).
1
Fundamentally wrong or misleading. Describes the wrong concept entirely, confuses it with something else, or gives advice that would cause data loss or errors.
from mlflow.genai.judges import make_judge tech_accuracy = make_judge( name="databricks_technical_accuracy", instructions=""" Evaluate if {{ outputs }} is technically accurate for the Databricks/Delta Lake question in {{ inputs }}. Score 1-5: 5 = Precise, correct, includes important caveats 4 = Mostly correct, minor omissions 3 = Partially correct, key concept present but incomplete 2 = Significant inaccuracy 1 = Fundamentally wrong or misleading Reason step-by-step before scoring. First identify the key claims in the response. Then verify each claim against known Databricks documentation. Finally assign a score based on the rubric above. """, model="databricks:/databricks-claude-sonnet-4" ) # Use exactly like a built-in judge results = mlflow.genai.evaluate( data=dataset, predict_fn=my_agent, scorers=[tech_accuracy, Correctness(), Safety()] )
Calibration loop

After running your custom judge, inspect the per-row rationales. If the judge scores a row 4/5 that you'd rate 2/5, the rubric needs revision — add the specific failure pattern to the appropriate level description. This iterative "run → inspect → refine" cycle is the calibration loop. A well-calibrated judge takes 3–5 iterations. Module 7 formalises this with Pearson correlation and Cohen's kappa measurement.

The model parameter determines both quality and cost. For pre-deployment quality gates, use a frontier model like databricks-claude-sonnet-4. For high-volume production monitoring, a smaller model reduces cost — but verify alignment doesn't drop below the 0.7 Pearson threshold first (Module 7).
4.5 · Hands-On Lab

Lab: Guidelines Judge for Compliance & Style Rules

Guidelines are the fastest way to enforce compliance, style, and policy requirements. Unlike make_judge() which uses a 1–5 rubric, Guidelines evaluates against a flat list of rules and returns a binary pass/fail verdict per rule. This makes it ideal for non-negotiable constraints where partial credit doesn't make sense.

Guidelines vs. make_judge() — when to use which

Dimension Guidelines make_judge()
Output type Binary pass/fail per rule Numeric rubric (1–5)
Input A list of rule strings A full prompt with template variables and rubric
Granularity Per-rule verdicts — one response can pass some and fail others Single aggregate score for the entire response
Setup effort Low — just write the rules Medium — prompt + rubric + calibration
Best for Policy compliance, format rules, safety, language Domain quality, nuanced technical accuracy

The decision is simple: if the answer to "did the response meet this criterion?" is unambiguously yes or no, use Guidelines. If it's "mostly" or "partially," you need a rubric-based judge via make_judge().

Patterns

Two Guidelines patterns

Pattern 1
Global Guidelines
Rules defined at judge instantiation time that apply identically to every row. Use for universal policies: "response must be in English", "must not disclose internal API keys", "must not recommend dropping tables."
Pattern 2
ExpectationsGuidelines
Per-row rules stored in the dataset's expectations field. Each row has its own compliance criteria. Use when domain experts define row-specific requirements — e.g., "this answer must mention ACID guarantees."
from mlflow.genai.scorers import Guidelines, ExpectationsGuidelines # ── Pattern 1: Global rules — same for every row ── compliance = Guidelines( name="compliance", guidelines=[ "Response must not recommend deleting production tables", "Response must be in English", "Must cite a Databricks documentation section for technical claims", ] ) # ── Pattern 2: Per-row rules from dataset expectations ── # Dataset row: {"inputs": {...}, "expectations": {"guidelines": ["Must mention ACID"]}} domain_rules = ExpectationsGuidelines() # ── Compose the FULL scorer suite from this module ── results = mlflow.genai.evaluate( data=dataset, predict_fn=my_agent, scorers=[ Correctness(), # built-in rubric judge (4.2) tech_accuracy, # custom make_judge (4.4) compliance, # global guidelines (4.5) domain_rules, # per-row guidelines (4.5) word_count, # code scorer (4.3) latency_ok, # code scorer (4.3) ] )
Full suite composition

The code above is the culmination of this module — a single evaluate() call running a built-in judge, a custom rubric judge, global Guidelines, per-row Guidelines, and two code scorers simultaneously. MLflow executes them concurrently via thread pools. This is the standard pattern for production evaluation: layer deterministic checks (code) with semantic assessment (judges) and compliance enforcement (guidelines).

Interpreting combined results — metric keys for CI gates

When you run a mixed scorer suite, the results object contains metrics from all types in a flat namespace. Understanding which metric keys correspond to which scorer type is essential for building CI gate thresholds and monitoring dashboards.

Metric key Scorer type Aggregation Typical CI gate
correctness/mean Built-in judge Mean of 1–5 scores ≥ 3.5
databricks_technical_accuracy/mean Custom judge Mean of 1–5 scores ≥ 3.0
safety/ratio_pass Built-in judge Fraction of "yes" = 1.0 (zero tolerance)
compliance/ratio_pass Guidelines Fraction all rules passed ≥ 0.95
word_count/mean Code scorer Mean word count Application-specific
latency_ok/ratio_pass Code scorer Fraction of True ≥ 0.90
