6.1 · Theory

Databricks AI Gateway — Architecture & Role in Evaluation

The AI Gateway is a centralised control plane that sits between your applications and every LLM endpoint. Think of it as a smart proxy — every request to any model passes through it, giving you a single enforcement point for policies, cost controls, and safety guardrails. In the context of evaluation, AI Gateway is the first line of defence: it catches violations in real time, before they ever reach your inference tables or scorers.

Four core capabilities

Capability 1
Rate limiting
Controls how many requests hit your model endpoints per minute or hour. Prevents runaway costs from sudden traffic spikes and ensures fair resource allocation across teams and applications.
Capability 2
Cost tracking
Logs token counts and associated costs per request, per model, per user. You get a unified cost view across all LLM providers — OpenAI, Anthropic, and Databricks Foundation Models — in a single pane.
Capability 3
Model routing
Directs traffic to different model endpoints based on rules. Route simple queries to a cheaper model and complex queries to a frontier model, or implement A/B testing across model versions.
Capability 4
Guardrails
Applies input and output filters that block requests in real time. PII detection, toxicity filtering, topic restriction, and custom safety rules — all enforced synchronously before the response reaches the user.
Architecture

Request flow through AI Gateway

Every request from your application passes through the Gateway's input guardrails first, then rate limiting and routing, then to the model. The response returns through output guardrails before reaching the user. Any guardrail violation can block the request entirely or warn (log and pass through).

Key concept

The two-layer defence model

This is the architectural insight that connects AI Gateway to the rest of the evaluation pipeline. Gateway guardrails and registered scorers serve fundamentally different purposes, operate at different speeds, and catch different failure modes.

Layer 1
Gateway guardrails
Synchronous · real-time · blocking
Layer 2
Registered scorers
Asynchronous · background · analytical
Gateway guardrails run inline with every request. If a user sends PII or the model returns something unsafe, the Gateway blocks it before it reaches the user. This is your safety net — fast, binary (pass/block), and synchronous. Think of it as the bouncer at the door: immediate rejection of clearly bad requests. Guardrails don't assess quality — they enforce policy. A response can pass all guardrails and still be factually wrong.
Registered scorers run in the background after the response has already been served. They evaluate quality dimensions — correctness, groundedness, compliance — using .register(name="...").start(). They don't block anything; they produce analytics. Think of them as the quality audit team: analysing patterns after the fact to spot degradation trends. Use these to detect when your model is getting worse, not to prevent individual bad responses.
Design principle

Gateway guardrails catch policy violations (things that should never happen). Registered scorers catch quality degradation (things that technically pass policy but aren't good enough). A response can sail through every guardrail and still score poorly on correctness — that's the scorer's job to detect.

Guardrail types and configuration modes

Guardrail Direction What it catches Mode
PII detection Input SSNs, credit card numbers, email addresses in queries BLOCK or WARN
Toxicity filtering Input Hate speech, harassment, and abusive language BLOCK or WARN
Topic restriction Input Off-domain queries that your agent shouldn't handle BLOCK or WARN
Safety filter Output Harmful, inappropriate, or dangerous model responses BLOCK or WARN
BLOCK rejects the request entirely and returns an error response. WARN logs the violation but lets the request through. Most teams use WARN during initial rollout to calibrate false-positive rates, then switch to BLOCK once confident in the guardrail's precision.
API

Enabling AI Gateway on a serving endpoint

from databricks.sdk import WorkspaceClient from databricks.sdk.service.serving import AiGatewayConfig, GuardrailParameters w = WorkspaceClient() w.serving_endpoints.put_ai_gateway( name="my-databricks-agent", guardrails=AiGatewayConfig( input_guardrail=GuardrailParameters( pii="BLOCK", # block PII in inputs toxicity="WARN", # log but allow toxic inputs (calibration phase) ), output_guardrail=GuardrailParameters( safety="BLOCK" # block unsafe model outputs ) ), inference_table_config={ # enable inference table (covered in 6.3) "enabled": True, "catalog_name": "main", "schema_name": "genai_eval", "table_name_prefix": "tutorial_agent" } )
6.3 · Theory

Inference Tables — Architecture & Schema

Inference Tables are the automatic audit trail for every request your served model handles. When enabled on a Model Serving endpoint, every request-response pair gets logged as a row in a Unity Catalog Delta table — no custom logging code required. Think of it as a flight recorder for your AI application: every call, every response, every trace, every token count — captured automatically and queryable with SQL or Spark.

How inference tables work

When a request hits your serving endpoint, the system does two things simultaneously: serves the response to the user, and writes a record to the inference table. This happens transparently — your application code doesn't change. For agents with MLflow tracing enabled, the table also stores the full trace JSON per request, meaning you can reconstruct every retrieval step, LLM call, and tool invocation after the fact.

Schema

The three auto-generated tables

When you enable inference tables on a serving endpoint, Databricks automatically creates three Delta tables in your specified Unity Catalog schema. Each serves a distinct purpose in the evaluation lifecycle.

Table 1
_request_response
The primary table. Every row is one request-response pair. Captures full request payload, full response, timestamp, latency, token count, and the databricks_request_id unique identifier. When tracing is enabled, also stores the complete MLflow trace JSON — letting you reconstruct every span after the fact.
primary
Table 2
_assessment_logs
Stores all feedback and scorer results attached to traces. When registered scorers run via .register().start(), their results land here. When humans submit feedback through the Review App or mlflow.log_feedback(), those ratings also appear here. The single source of truth for "how good was this response?"
feedback
Table 3
_payload_request_logs
Detailed request metadata: HTTP headers, client information, routing decisions, and Gateway-level annotations (e.g., whether a guardrail was triggered). Primarily useful for debugging and compliance auditing, not for evaluation workflows.
metadata

What gets captured per request

Every row in the main _request_response table includes these fields. The trace column is the most important for evaluation — it gives you the full execution graph of every agent call.

Field Type What it contains
databricks_request_id String Unique identifier — the key to look up the full trace later
request JSON Full request payload your app sent to the endpoint
response JSON Full response the endpoint returned
timestamp_ms Long When the request arrived (millisecond precision)
execution_time_ms Long End-to-end latency from receipt to response
status_code Int HTTP status (200, 429, 500, etc.)
token_count Struct Input and output token counts for cost tracking
trace JSON Full MLflow trace JSON (when tracing is enabled)
Setup

How to enable inference tables

There are two paths. If you deploy your agent using mlflow.deploy(), inference tables are enabled automatically — no extra configuration needed. For existing endpoints, you enable them via the AI Gateway config shown in section 6.1. Once enabled, logging starts immediately. The tables are standard Unity Catalog Delta tables, so they inherit all UC governance: access controls, lineage tracking, and audit logging.

For the inference table to capture full trace JSON (not just request/response), your serving endpoint must have MLflow tracing enabled. Either set ENABLE_MLFLOW_TRACING=True in your endpoint config, or deploy using mlflow.deploy() which enables it by default. Without tracing, you get request/response logging but lose the ability to inspect individual spans.
Pattern

The production-to-evaluation closed loop

The real power of inference tables is that they create a closed loop between production and development. Production traffic feeds the eval dataset, which improves scorers, which improve the agent, which changes production traffic. This is the canonical LLMOps feedback cycle.

Step 1 — Accumulate production data
Your serving endpoint handles real user traffic. Every request, response, and trace gets logged automatically to the inference table. No code changes required.
auto
Step 2 — Query and filter
Use Spark SQL to filter the table for interesting cases: recent requests, failed requests, low-scoring requests, or specific user segments. The databricks_request_id lets you look up the full trace.
spark.table()
Step 3 — Build eval dataset from real traffic
Extract traces from the inference table and add them to a UC-backed evaluation dataset using eval_dataset.merge_records(traces=...). Far more representative than hand-curated examples.
merge_records()
Step 4 — Evaluate in dev or CI
Run mlflow.genai.evaluate() against the production-sourced dataset during development or as a CI quality gate. If scores drop below thresholds, block deployment.
evaluate()
Step 5 — Continuous monitoring
Meanwhile, registered scorers run async against a sample of production traces, writing results to _assessment_logs. Live quality trends with no manual intervention.
register().start()
# Query the inference table and build an eval dataset from production inf_table = spark.table("main.genai_eval.tutorial_agent_request_response") from pyspark.sql import functions as F # Filter: last 24h, successful, non-gateway-blocked recent_reqs = ( inf_table .filter(F.col("timestamp_ms") > (F.unix_timestamp() - 86400) * 1000) .filter(F.col("status_code") == 200) .orderBy("timestamp_ms", ascending=False) .limit(50) ) # Get request IDs → look up traces → add to eval dataset req_ids = [r.databricks_request_id for r in recent_reqs.collect()] traces = mlflow.search_traces( filter_string=f"tags.`mlflow.databricks.requestId` IN ({','.join(repr(i) for i in req_ids)})" ) eval_dataset.merge_records(traces=traces)
6.5 · Theory

Production Monitoring with Registered Scorers

In Modules 3 and 4 you used scorers inside mlflow.genai.evaluate() — that's the dev/CI path. Now those same scorers graduate to production monitoring. The mechanism is .register(name="...").start(), which tells MLflow to continuously evaluate a sample of live production traces in the background. No code changes to your agent, no latency impact on the user — the scorer runs asynchronously after the response is served.

The register → start lifecycle

Registering a scorer is a two-step process. First, .register(name="unique_name") creates a named scorer entity in your MLflow experiment. Then .start(sampling_config=...) activates it — the scorer begins polling for new traces and evaluating them at the configured sample rate. Results appear as feedback objects on the trace, visible in the MLflow UI's Scorers tab.

Constraint

Which scorers can be registered for production

This is a critical constraint you must design around from the start. Only LLM judges support .register().start(). Code-based @scorer functions cannot be registered for continuous monitoring — they only run inside evaluate() during dev and CI.

Scorer type evaluate() register().start() Examples
Built-in judges Yes Yes Correctness(), Safety(), RelevanceToQuery()
Custom judges Yes Yes make_judge() with custom rubric and prompt
Guidelines Yes Yes Guidelines(name="compliance", guidelines=[...])
Code-based scorers Yes No @scorer — latency check, word count, JSON format
Design your scorer suite with this boundary in mind. Code-based checks like latency and format validation must be covered during dev/CI evaluate() runs. Only semantic quality, safety, and compliance can be monitored continuously in production.

Configuring sampling rates

Every registered scorer evaluates a sample of production traces — not all of them. The ScorerSamplingConfig controls the rate and optionally filters which traces to evaluate. Smart sampling is essential: running 5 LLM judges at 100% on 10,000 daily traces means 50,000 extra LLM calls purely for evaluation.

Safety
High sample rate (100%)
Safety scorers should evaluate every trace. A single unsafe response is a critical failure. The cost of running a safety judge on every trace is trivially small compared to the reputational and legal cost of missing one violation.
Quality
Medium sample rate (30–50%)
Quality judges like Correctness detect trends, not individual failures. A 30% sample is statistically sufficient to detect a 5-point drop in mean score. Higher rates cost more but don't meaningfully change alerting sensitivity.
Compliance
Full rate with filter (100%)
Compliance rules via Guidelines are cheap to run and critical for regulated industries. Run at 100% but use filter_string to scope to specific environments or user segments.
API

Registering scorers — the full pattern

from mlflow.genai.scorers import Safety, Correctness, Guidelines, ScorerSamplingConfig # ── Safety: 100% sample rate — no unsafe response should go unscored ── safety_monitor = Safety().register(name="prod_safety_v1") safety_monitor.start( sampling_config=ScorerSamplingConfig(sample_rate=1.0) ) # ── Correctness: 30% sample — enough to detect quality trends ── correctness_monitor = Correctness().register(name="prod_correctness_v1") correctness_monitor.start( sampling_config=ScorerSamplingConfig(sample_rate=0.3) ) # ── Compliance: 100% but filtered to prod environment traces ── compliance_monitor = Guidelines( name="compliance", guidelines=[ "Must not recommend dropping production tables", "Response must be in English", ] ).register(name="prod_compliance_v1") compliance_monitor.start( sampling_config=ScorerSamplingConfig( sample_rate=1.0, filter_string="tags.environment='prod'" ) ) # ── Stop a scorer when no longer needed ── # correctness_monitor.stop()
Scorer versioning

Each registered scorer has a unique name scoped to the experiment. When you change a scorer's prompt or rules, register it with a new name (e.g., "prod_correctness_v2") rather than overwriting the old one. This preserves historical trend data and lets you run old and new versions side by side during a transition period.

Full picture

The complete production evaluation stack

Zooming out, here is how every piece from this module connects into a single production system. AI Gateway handles synchronous safety. Inference Tables capture the audit trail. Registered scorers provide continuous quality analytics. And the closed loop feeds production data back into your eval datasets.

The complete picture

Three layers work together: AI Gateway (synchronous safety — blocks violations before they enter the pipeline), Inference Tables (storage — the audit trail and data source for everything downstream), and Registered Scorers (asynchronous quality — continuous background monitoring that detects degradation trends). The closed loop feeds production data back into eval datasets, which improve scorers, which improve the agent.
