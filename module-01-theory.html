1.1 · Theory

Why Evaluating LLMs and Agents is Fundamentally Different

Traditional software testing works on a simple premise: a function given the same input must always return the same output. You write a test, it passes or fails, and that result holds forever. Language models break every assumption this premise rests on.

Problem 1

The non-determinism problem

Every time you call an LLM with the same prompt, temperature sampling means you get a different token sequence. The same exact question can produce a brilliant answer, a mediocre one, or a hallucinated one — on three consecutive calls. This makes unit testing not just hard but conceptually wrong: a test that passes once tells you nothing about what the model will return in production.

Key insight

You cannot test an LLM — you can only measure it statistically. Evaluation requires running many examples and looking at distributions of scores, not individual pass/fail results.

Problem 2

Classical metrics completely fail

For decades, NLP relied on BLEU, ROUGE, and exact-match metrics. These work by counting word overlaps between a model's output and a "reference" answer. They were designed for tasks like translation and summarisation where the reference is fixed. They are actively harmful for open-ended generation evaluation.

Metric failure demo — Q: "What does the VACUUM command do in Delta Lake?"
Reference
"VACUUM removes old data files no longer referenced by Delta table versions, freeing storage."
ground truth
Answer A
"VACUUM removes old versions, files, and data tables from the Delta directory and cleans referenced records." (factually wrong — claims it removes referenced data)
BLEU 0.71 High word overlap
— rated good
Answer B
"It cleans up stale files that Delta no longer tracks, reclaiming disk space without affecting current data." (semantically correct)
BLEU 0.14 Low word overlap
— rated bad

BLEU rewards copying the reference's vocabulary, not getting the answer right. It will systematically rate a wrong answer higher than a correct one that uses different words — exactly the inverse of what you need.

Problem 3

Correctness is a spectrum, not binary

For most real-world LLM tasks, there is no single correct answer — there is a continuous spectrum from completely wrong through partially correct to expert-level. A binary pass/fail test cannot capture "mostly right with a minor omission." Evaluation must work on a scoring rubric.

Score Answer quality Binary test verdict Reality
1 / 5 Fundamentally wrong — describes the wrong concept entirely FAIL Correctly rejected
2–3 / 5 Partially correct — right concept, missing key details or edge cases FAIL Useful output, incorrectly rejected
4 / 5 Mostly correct — complete answer, minor caveats missing FAIL Good answer, incorrectly rejected
5 / 5 Expert-level — precise, complete, includes important nuance PASS Correctly accepted
A binary test only distinguishes "exactly right" from everything else. For open-ended generation, this means 75% of genuinely useful outputs get incorrectly rejected.
Problem 4

Agents add three independent failure layers

A standalone LLM is hard enough to evaluate. An agent — one that retrieves documents, calls tools, and reasons across multiple steps — adds entirely new failure modes. Each layer can fail independently, and a correct final answer can mask failures at earlier stages.

Layer 1 — Retrieval
Vector search, document fetch, context assembly. Failure: wrong documents fetched, relevant documents missed (recall failure), or too many irrelevant documents included (precision failure).
RetrievalRelevance scorer
Layer 2 — Reasoning & tool use
Decide which tools to call, in what order, with what parameters. Failure: redundant tool calls, wrong tool selected, wrong arguments, infinite retry loops without new information.
ToolCallEfficiency scorer
Layer 3 — Generation
Synthesise the final answer from retrieved context. Failure: hallucination (claims not in context), ignoring relevant retrieved facts, tone or format violations.
Groundedness + Correctness
Critical consequence

An agent can produce a correct final answer via a completely broken pipeline — retrieving wrong docs, making 8 redundant tool calls, and then guessing the right answer from training data. A single end-to-end quality score would show green while the system is fundamentally broken. You need layer-specific scorers.

Problem 5

The evaluation triangle — you can't optimise all three axes

Every evaluation decision is a tradeoff between three competing objectives. No single approach wins on all three simultaneously — which is why mature evaluation systems use all three paradigms in combination.

Q
Quality
How accurately the evaluation reflects true model quality, including nuance, context, and domain knowledge.
$
Cost
Monetary and latency cost per evaluation call. LLM judges cost tokens; human review costs expert time.
∞
Scale
Volume of traces that can be evaluated. Production systems may generate 10,000+ requests per day.
Human review: highest quality, very high cost, very low scale.   LLM judges: good quality, moderate cost, high scale.   Code heuristics: deterministic quality, near-zero cost, 100% scale.
MLflow's answer: use all three in a tiered pipeline.

The three evaluation paradigms

No single paradigm solves all five problems above. MLflow 3 is designed around all three working together — you define scorers once and they run at every stage.

What "good" looks like — the five quality dimensions

Rather than a single "quality" score, production evaluation decomposes quality into five independently measurable dimensions. Each maps to a specific MLflow scorer.

Correctness
Is the factual content accurate? Does it match expected facts?
Relevance
Does the response actually address what the user asked?
Groundedness
In RAG: does the answer stay within retrieved context?
Safety
Is the response free of harmful, toxic, or policy-violating content?
Efficiency
For agents: did it reach the answer without redundant steps?
Why decompose?

A single "quality" score hides root causes. If your agent has a Safety problem, you fix the system prompt. If it has a Groundedness problem, you fix the retrieval or context window. If it has an Efficiency problem, you reduce tool calls. Each dimension points to a different fix.

1.2 · Theory

MLflow 3 GenAI Stack — The Full Map

MLflow 3 introduces a dedicated mlflow.genai module built specifically around the evaluation problems described above. Rather than a single evaluation function, it's a connected system of objects that persist across your entire model lifecycle — from first prototype through production monitoring.

Core objects

Five first-class objects form the backbone of the system. Everything else is built on top of these.

Object 1
Experiment
The top-level container for all work on one application. All runs, traces, datasets, and scorers for a given agent live under a single experiment. Created with mlflow.set_experiment().
Object 2
Run
One execution of mlflow.genai.evaluate(). Stores the dataset used, the model version tested, all per-row scores, aggregate metrics, and links to every trace generated. Your unit of "one evaluation attempt."
Object 3
Trace
One execution of your agent — a hierarchical tree of spans capturing every LLM call, tool call, retrieval step, and their inputs/outputs. Generated automatically via mlflow.openai.autolog() or @mlflow.trace.
Object 4
Logged model
A versioned snapshot of your agent code registered in Unity Catalog. The model_id parameter in evaluate() links evaluation runs to specific model versions — this is how you get lineage.
Object 5
Scorers
The evaluation functions — code-based @scorer decorators, built-in LLM judges, and custom make_judge() judges. Defined once and shared across dev, CI gate, and production monitoring without modification.

The evaluation pipeline

Every call to mlflow.genai.evaluate() follows the same five-stage pipeline. Understanding this flow is essential — it determines how you structure your datasets, how your agent gets called, and how scores end up on traces.

Stage 1
Evaluation dataset
Rows of inputs + optional expectations. Can be a list, Pandas/Spark DataFrame, or UC-backed EvaluationDataset. MLflow iterates over rows and calls predict_fn for each one.
UC Delta table
Stage 2
predict_fn → Trace
Your agent function (or a deployed endpoint via to_predict_fn()). MLflow unpacks each row's inputs dict as keyword arguments, calls your function, and captures one trace per row. The trace is the unit all scorers operate on.
@mlflow.trace
Stage 3
Scorers run in parallel
All scorers in your scorers=[...] list run concurrently against each trace using thread pools. MLflow controls parallelism via MLFLOW_GENAI_EVAL_MAX_WORKERS and MLFLOW_GENAI_EVAL_MAX_SCORER_WORKERS.
concurrent
Stage 4
Feedback objects attached to traces
Each scorer returns a Feedback object containing a value (score), rationale (written justification), and source (who scored: CODE / LLM_JUDGE / HUMAN). These attach to the trace as assessments, visible in the MLflow UI.
per-row
Stage 5
Aggregate run metrics
MLflow aggregates per-row scores into run-level metrics: correctness/mean, safety/ratio_pass, etc. These are the values you threshold against in your CI gate. The run also carries a model_id link to the agent version being tested.
run metrics

How a scorer travels: dev → CI gate → production monitoring

The central architectural principle of MLflow 3 is that the same scorer definition runs in all three environments. You write a Correctness() judge once in a notebook. That exact same object becomes the CI gate threshold check in your Databricks Workflow. And then Correctness().register(name="prod_v1").start() turns it into continuous background monitoring against live production traces.

Environment How the scorer runs What it operates on Scorer types supported
Development mlflow.genai.evaluate(scorers=[...]) UC eval dataset — curated, synthetic, or from traces All types code + LLM + Guidelines
CI gate evaluate() + threshold check + dbutils.notebook.exit() Same UC dataset against the deployed endpoint via to_predict_fn() All types code + LLM + Guidelines
Production monitoring scorer.register(name).start(sampling_config) Live production traces sampled at configured rate LLM judges only code scorers not supported
Code-based @scorer functions are not supported in production monitoring — only LLM judges (built-in, make_judge(), and Guidelines). This means your latency check and JSON format check run in dev and CI, but cannot be registered for continuous monitoring.

Databricks additions over OSS MLflow

The mlflow.genai module is open-source and runs anywhere. Databricks adds four production infrastructure components that are exclusive to the managed platform.

Databricks exclusive
Unity Catalog
Eval datasets stored as versioned Delta tables. Every dataset, model, endpoint, inference table, and evaluation run has lineage tracked end-to-end in one governance layer.
Databricks exclusive
AI Gateway
A proxy that sits in front of all LLM endpoints. Enforces PII blocking, toxicity filtering, topic restrictions, rate limiting, and cost tracking synchronously before requests reach your agent.
Databricks exclusive
Inference Tables
Auto-captured Delta tables in UC storing every production request/response plus the full MLflow trace JSON per call. The source of truth for production evaluation datasets and monitoring.
Databricks exclusive
Review App
A no-code UI where domain experts browse traces and attach human feedback. Labels flow into the same Feedback schema as automated scorers, enabling direct comparison between human and automated quality signals.

MLflow 3 vs MLflow 2 — what changed

MLflow 3 is not a minor update — it's a redesign of how evaluation fits into the model lifecycle. The core shift: evaluation moves from a one-off measurement tool into a continuous quality system.

Capability MLflow 2 MLflow 3
Evaluation API mlflow.evaluate() — generic, model-type agnostic mlflow.genai.evaluate() — GenAI-specific with predict_fn, scorers
Evaluation datasets Plain DataFrames, no persistence UC-backed EvaluationDataset — versioned, governed, shared
Custom scorers EvaluationMetric class — verbose, complex @scorer decorator — simple, supports trace access, Feedback objects
Production monitoring Not available .register().start(sampling_config) — continuous, sampled
Human feedback Not available mlflow.log_feedback() + Review App — same schema as automated scores
Tracing Basic span logging Full OpenTelemetry-compatible trace tree, searchable, with autolog for 10+ frameworks
Model versioning tie-in Manual tracking model_id in evaluate() — automatic lineage from eval run → model version
