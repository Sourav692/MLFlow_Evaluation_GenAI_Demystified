7.1 · Theory

Human Feedback in MLflow — Review App & log_feedback()

Automated judges are fast, scalable, and cheap — but they are not automatically trustworthy. Without a steady stream of human signal flowing back into the system, judges drift from real domain standards, and there's no way to know which side of the gap your eval scores are on. Human feedback is the ground truth that keeps the whole evaluation pipeline honest.

What human feedback unlocks

It calibrates automated judges (the input to alignment in §7.3), surfaces failure modes that scorers miss, builds trust with stakeholders before production rollout, and continuously improves the eval dataset with real-world signals. Skip it, and your release decisions ride on numbers that may have no connection to user experience.

Two collection paths, one storage model

MLflow gives you two distinct ways to capture feedback, designed for two very different audiences. Both write to the same underlying schema, so downstream queries don't care which path the data came through. Click each path to inspect it.

Architecture · how feedback lands on a trace

The Feedback object — five fields, one schema

Every piece of feedback — a Review App label, a 👍 from a customer, an LLM judge verdict, a code scorer result — becomes a Feedback object with the same five fields. This is why the MLflow UI can show human and automated scores side by side without any extra plumbing.

Field 1 · Required
name
What is being assessed. Examples: "user_helpful", "correctness", "safety". Stable across sources — the same name from a judge and a human is directly comparable.
Field 2 · Required
value
The verdict. Type matches the assessment: bool for thumbs-style, int 1–5 for rubrics, float for similarity scores, str for category labels.
Field 3 · Required
source
An AssessmentSource with source_type = "HUMAN", "LLM_JUDGE", or "CODE". This is the field the UI uses to group and contrast scores.
Field 4 · Recommended
source_id
Who or what produced the assessment — a user ID, judge name, or scorer version. Lets you filter by reviewer cohort or judge generation.
Field 5 · Recommended
rationale
Free-text justification. For judges: the chain-of-thought. For humans: the comment they left. Critical for debugging why a score landed where it did.
Why the unified schema matters

Because all sources land in the same shape, you can write one query that returns the human score, the judge score, and the code-scorer score for every trace — and compute the gap between them. That gap is the input to judge alignment in §7.3.

7.3 · Theory

Judge Alignment — Calibrate Automated Scores to Human Judgment

LLM judges are scalable, but they are not automatically calibrated to your domain. A correctness judge might confidently rate a response 3.5 / 5 when an expert reviewer would have rated it 2.0 / 5. Two prompt versions might both score 4.2 from the judge while being objectively very different to a human. The drift compounds silently as your models and prompts evolve.

Treat any judge score as a directional opinion until you've measured its alignment. High judge scores create false confidence — you think quality is good when it isn't. Release gates that ride on uncalibrated judges will eventually let through regressions that humans would have caught.

How alignment works in MLflow

Mechanically, alignment is straightforward: you feed the judge examples of how humans actually rated similar traces, embedded as few-shot anchors directly in the judge prompt. The judge learns to map "this kind of response" → "this kind of score" the way humans do. You're not retraining the LLM — you're giving it calibration data in its context.

Input 1
Gold set
30 to 100 traces with human ratings collected via the Review App or log_feedback(). Used to construct the few-shot anchors in the judge prompt.
Input 2
Target judge
The judge whose calibration you want to improve — a make_judge() custom judge or a built-in. Alignment edits the prompt, not the underlying model.
Input 3
Holdout set
A separate batch of human-labelled traces, never shown to the judge. Used only to measure alignment. Without it, you'd overfit to your anchors.
The gold set (used to build the prompt) and the holdout set (used to measure alignment) must be disjoint. If you measure on the same traces you anchored on, correlation will always look great — and your CI gate will be lying to you.

Measuring alignment: Pearson + Cohen's κ

Two complementary statistics, because neither is sufficient alone. Pearson captures whether the judge moves in the same direction as humans — great for catching ranking quality even when absolute scores are biased. Cohen's kappa measures absolute agreement on discrete buckets, corrected for chance — catches the case where the judge correlates but is systematically off.

Statistic 1
Pearson correlation (r)
Linear correlation between judge scores and human scores across the holdout. Best when scores are continuous or rubric-style (1–5). Computed with scipy.stats.pearsonr.
Statistic 2
Cohen's kappa (κ)
Chance-corrected agreement on discrete or categorical scores. Best when ratings are pass/fail or bucketed. Computed with sklearn.metrics.cohen_kappa_score.
Statistic Below 0.4 0.4 – 0.7 Above 0.7
Pearson r Unusable — judge doesn't track human direction Directional only — risky for gating decisions Trustworthy — safe for release gates
Cohen κ Near-random agreement Moderate — keep iterating Strong agreement
The practical bar

Don't trust an automated score for release gates until both Pearson and kappa clear 0.7 against your human holdout. If only one passes, you're either catching ranking but not magnitudes, or catching magnitudes but not ordering — both fail in real CI use.

The calibration loop

Alignment is iterative — you almost never hit threshold on the first try. The loop runs until the holdout numbers clear the gate. Steps 2 and 5 are where most iteration happens: refine the rubric, add more diverse anchors, balance the rating distribution, retry.

1
Collect human labels
30–100 traces rated by experts via Review App. Balance the rating distribution — all-5s teaches the judge nothing.

2
Anchor in judge prompt

3
Score the holdout
Run the anchored judge against the held-out, human-labelled traces. Collect parallel arrays of judge and human scores.
evaluate()
4
Measure correlation + kappa
Compute pearsonr(judge, human) and cohen_kappa_score(judge, human) on the parallel arrays.
scipy/sklearn
5
Gate check — and iterate
If both ≥ 0.7 → judge is trusted for CI. If not → refine rubric, add edge-case anchors, rebalance distribution, jump back to step 2.

import mlflow, numpy as np from scipy.stats import pearsonr from sklearn.metrics import cohen_kappa_score from mlflow.genai.judges import make_judge # 1. Pull the gold set: traces that already have human feedback gold = mlflow.search_traces( filter_string="assessments.name = 'human_correctness' AND tags.split = 'gold'" ) # 2. Define a judge with few-shot anchors built from the gold set correctness_judge = make_judge( name="correctness_aligned_v1", instructions=""" Rate {{ outputs }} for correctness given {{ inputs }}. Score 1-5. Examples below show how human reviewers scored similar cases: Example 1 (Human score: 2) — ... Example 2 (Human score: 4) — ... Example 3 (Human score: 1) — ... """, model="databricks:/databricks-claude-sonnet-4" ) # 3. Score the holdout (disjoint from gold) holdout = mlflow.search_traces(filter_string="tags.split = 'holdout'") judge_scores = [correctness_judge(trace=t).value for t in holdout] human_scores = [t.assessments["human_correctness"].value for t in holdout] # 4. Measure alignment r, _ = pearsonr(judge_scores, human_scores) k = cohen_kappa_score(np.round(judge_scores).astype(int), np.round(human_scores).astype(int)) print(f"Pearson r = {r:.3f} | Cohen kappa = {k:.3f}") # 5. Gate if r >= 0.7 and k >= 0.7: print("Judge aligned — safe to use in CI gate") else: print("Refine rubric, add edge cases, repeat")

Common pitfalls

Pitfall 1
Aligning and measuring on the same set
Correlation will always look strong because the judge has seen the answers. Alignment must be measured on a strictly held-out set the judge has never been anchored on.
Pitfall 2
Too few human labels
Under 20 examples produces unstable correlation estimates that swing wildly with one outlier. Aim for 50+ on the holdout, more if scores are noisy.
Pitfall 3
Skewed gold-set distribution
If 90% of human ratings are 5, correlation can look high but the judge has only learned to output 5. Force balance across the rating range when sampling.
Pitfall 4
Aligning once and forgetting
Alignment drifts when the underlying model, prompt template, or app changes. Re-measure after every meaningful change and on a quarterly cadence.
Pitfall 5
Ignoring κ when Pearson is high
A judge can correlate well in direction while being absolutely biased — e.g., always rating one point higher than the human. Check both statistics every time.
