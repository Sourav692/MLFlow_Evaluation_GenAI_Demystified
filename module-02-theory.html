2.1 · Theory

Evaluation Dataset Architecture in MLflow 3

In Module 1 you learned that LLM evaluation requires running many examples and looking at score distributions. The evaluation dataset is where those examples live. MLflow 3 treats datasets as first-class, governed objects — not throwaway CSVs loaded into a notebook. This architectural choice enables versioning, lineage, cross-team sharing, and reproducible evaluations across the entire model lifecycle.

Schema

The four-field dataset schema

Every evaluation dataset in MLflow 3 follows a strict four-field schema. This schema is the universal contract — every scorer, every judge, and the evaluate() function itself expects data in this exact shape. The consistency means any dataset from any source plugs directly into the evaluation pipeline with zero transformation.

Field 1 · Required
inputs
A dictionary containing the data sent to your agent. The keys must exactly match your predict_fn parameter names. For example: {"question": "What is Delta Live Tables?"} pairs with def predict_fn(question: str).
Field 2 · Optional
outputs
The agent's generated response. Can be pre-populated if you've already run the agent, or left empty for evaluate() to fill at runtime via predict_fn. Pre-populated outputs skip the predict step — useful for evaluating cached responses.
Field 3 · Optional
expectations
Ground truth that judges use to score the output. The most common sub-field is expected_facts, which Correctness compares against the agent's response. Example: {"expected_facts": ["declarative pipeline", "streaming"]}.
Field 4 · Optional
tags
Metadata for filtering, grouping, and organizing rows. Use tags to slice evaluation results by category, difficulty level, or domain — for example {"domain": "delta_lake", "difficulty": "hard"}.
Critical contract

The keys inside inputs must exactly match the parameter names of your predict_fn. If your dataset has inputs.question, your function signature must be def predict_fn(question: str). A mismatch causes a silent failure — MLflow will not raise an error, but your agent won't receive the data it expects. Always verify this mapping first.

Sources

Three sources for evaluation data

Where your evaluation data comes from determines what it tests. MLflow 3 supports three distinct sourcing strategies — each one covers different quality angles, and a mature evaluation system uses all three in combination.

Source 1 — Hand-curated
Domain experts write specific Q&A pairs targeting known edge cases, compliance requirements, and critical business scenarios. These are high-quality but expensive rows — you'll typically have 10–30 of them. They're essential for regression testing: ensuring your agent never breaks on cases you've already identified as important.
merge_records(records=[...])
Source 2 — From production traces
Real user interactions extracted from your serving endpoint's trace log. Use mlflow.search_traces() to retrieve recent successful traces, then add them to your dataset. This gives you the most realistic evaluation data — real questions from real users — but ground truth labels must be added manually or via the Review App.
merge_records(traces=[...])
Source 3 — Synthetically generated
Use generate_evals_df from Databricks to auto-generate diverse Q&A pairs from your documentation. The tool analyses your documents, identifies topics, and creates questions at varying difficulty levels — covering edge cases and phrasings you'd miss manually. Expected answers are grounded in the source text.
generate_evals_df(docs=[...])

Recommended strategy

Start with synthetic generation to get broad coverage quickly (50 rows), add 10–20 hand-curated rows targeting specific failure modes you care about, and continuously enrich from production traces as real usage patterns emerge. The dataset is a living artifact — not a static file written once.

The dataset lifecycle

Evaluation datasets in MLflow 3 follow a continuous lifecycle. New rows flow in from all three sources, get versioned in Unity Catalog, feed into evaluate() calls, and — critically — evaluation failures get added back as new test cases. This closed loop is what separates a production evaluation system from one-off testing.

Governance

Why Unity Catalog–backed datasets matter

The UC-backed EvaluationDataset is the centerpiece of the architecture for enterprise teams. Four properties distinguish it from simply loading data into a Pandas DataFrame.

Property 1
Automatic versioning
Every merge_records() call creates a new immutable version backed by Delta Lake time travel. You can always reproduce a past evaluation by pinning to a specific version — essential for audit trails and debugging regressions months later.
Property 2
End-to-end lineage
Unity Catalog tracks which model version was evaluated against which dataset version, which scorers were used, and what results were produced. This chain of evidence is critical for regulated industries and enterprise compliance.
Property 3
Sharing & governance
Datasets live in a three-level namespace (catalog.schema.name). Access controls follow UC's permission model — one team can own the golden eval set while others consume it read-only, with full ACL enforcement.
Property 4
Experiment binding
Each dataset is bound to an MLflow experiment namespace. Evaluation runs, traces, and scorer results are all linked in a single lineage chain — one pane of glass for all quality data about an application.
The experiment binding means one dataset per experiment namespace. If you need separate datasets for different evaluation scenarios (e.g., regression vs. stress testing), create separate experiments or use tags within a single dataset to segment them.
2.4 · Theory + Lab

generate_eval_df — Synthetic Dataset Generation

Humans are bad at imagining all the ways users will ask questions. We write clean, obvious queries and miss the messy, ambiguous, edge-case phrasings that cause real production failures. generate_evals_df uses an LLM to systematically generate diverse evaluation pairs from your source documents — covering topics and phrasings you'd never think to write manually.

Pipeline

The internal generation pipeline

The function reads your documentation, analyses it for distinct topics and concepts, generates questions at varying complexity levels, and produces expected answers grounded in the source text. The output is a Spark DataFrame already in the MLflow evaluation schema — ready for evaluate() with zero transformation.

Your documents
A list of document strings or a Unity Catalog Volume path. Each chunk should be a self-contained passage (a few paragraphs), not an entire book.

Key parameters

from databricks.agents.evals import generate_evals_df # Your documentation chunks (from Delta Lake docs, UC docs, etc.) docs = [ "Delta Lake is an open-source storage layer that provides ACID transactions...", "VACUUM removes files no longer referenced by a Delta table...", "Z-ordering optimizes data layout for multi-dimensional clustering...", ] # Generate synthetic eval dataset synthetic_eval_df = generate_evals_df( docs=docs, num_evals=20, # Q&A pairs to generate (50-100 for prod) agent_description="A Q&A assistant for Databricks and Delta Lake topics" ) synthetic_eval_df.display() # inspect in Databricks notebook

The num_evals parameter is your primary lever for balancing coverage against cost. Each eval requires LLM calls to generate, so 100 evals costs roughly 5× what 20 evals costs. During development, 10–20 is sufficient for quick iteration; for production use, 50–100 gives enough diversity to catch non-obvious failure modes.

The agent_description parameter shapes the style and scope of generated questions. A vague description like "a helpful assistant" produces generic questions. A specific description like "A Databricks support agent answering questions about Delta Lake maintenance and Unity Catalog governance using official documentation" produces targeted, domain-relevant questions that actually test your agent's capabilities.

What makes synthetic generation valuable

The real power isn't automation — it's the diversity of questions that humans typically miss. Consider hand-curating an eval dataset about Delta Lake: you'll write "What is Delta Lake?" and "How does VACUUM work?" But real users ask things like "why is my table getting bigger even after deletes?", "can I undo a VACUUM?", or "what happens if two writers conflict?" These are the questions that reveal agent weaknesses.

Practical guidelines

Review before using. Synthetic generation isn't perfect — you may get duplicate questions, questions outside your domain, or questions too easy to be useful. Always run synthetic_eval_df.display() and scan the output. Remove weak rows and note topic gaps to fill manually.

Persist into a UC dataset. The output of generate_evals_df is an in-memory Spark DataFrame. To make it durable and versioned, merge it into your UC-backed evaluation dataset:

eval_dataset = mlflow.genai.datasets.get_dataset("main.genai_eval.tutorial_eval_v1") eval_dataset.merge_records(data=synthetic_eval_df) # Now synthetic evals are versioned alongside hand-curated rows

Regenerate as docs evolve. When you add new documentation or update existing content, re-run generate_evals_df to catch questions about the new material. A stale evaluation dataset misses new failure modes.

eval_data = [ {"inputs": {"question": "What is Delta Live Tables?"}, "expectations": {"expected_facts": ["declarative pipeline", "streaming"]}}, ] results = mlflow.genai.evaluate( data=eval_data, predict_fn=my_agent, scorers=[Correctness()] )

The fastest way to get an evaluation running. Write test cases directly in a notebook cell, pass them to evaluate(), and get results in seconds. Nothing is saved — if the kernel restarts, data is gone. Use when iterating on a prompt with 5–10 test cases, running a one-off sanity check, or debugging a specific failure with a minimal reproduction case.

import pandas as pd, json eval_df = pd.DataFrame([ {"question": "What is Delta Live Tables?", "expected_facts": '["declarative pipeline", "streaming"]'}, ]) # Transform into MLflow schema eval_data = [ {"inputs": {"question": row["question"]}, "expectations": {"expected_facts": json.loads(row["expected_facts"])}} for _, row in eval_df.iterrows() ]

When your evaluation data already exists as a CSV or Excel export, Pandas is the natural bridge. Handles hundreds to low thousands of rows, but no built-in versioning. Use when migrating from a legacy system, collaborating with non-engineering stakeholders who maintain eval cases in spreadsheets, or when you need Pandas groupby/pivot analysis on results.

# Load from a Delta table in Unity Catalog eval_spark_df = spark.table("main.genai_eval.curated_eval_cases") # Or filter a large pool down to a focused eval set eval_spark_df = ( spark.table("main.genai_eval.all_eval_cases") .filter("difficulty = 'hard'") .filter("domain = 'delta_lake'") .limit(200) )

The right choice when evaluation data exceeds driver memory or already lives in a Delta Lake table. Gives you full Spark manipulation — filtering, sampling, joining with other lakehouse tables — before passing to evaluate(). You get Delta Lake time travel for historical queries, but you're managing versioning yourself rather than through MLflow's dataset API.

import mlflow.genai.datasets # Create (one-time setup) eval_dataset = mlflow.genai.datasets.create_dataset( name="main.genai_eval.tutorial_eval_v1" ) # Add rows — each call creates a new version eval_dataset = eval_dataset.merge_records(records=[ {"inputs": {"question": "What is Delta Live Tables?"}, "expectations": {"expected_facts": ["declarative pipeline"]}}, ]) # Retrieve by name (from any notebook) eval_dataset = mlflow.genai.datasets.get_dataset( "main.genai_eval.tutorial_eval_v1" )

The only format purpose-built for evaluation workflows. Wraps a Delta table with MLflow-specific APIs for creating, versioning, and retrieving data. Every merge_records() creates an immutable version. The dataset binds to an MLflow experiment, linking runs, traces, and results in a single lineage chain. Use for any evaluation that matters — CI gates, production baselines, cross-team datasets.

The maturity journey

Most teams follow the same progression. List of dicts during early development when you're iterating on prompts. Pandas as an intermediate step when loading from CSV or doing ad-hoc analysis. Spark DataFrame when scale demands it or the data engineering team maintains the eval table. UC-backed dataset once you need governance, versioning, and cross-team sharing — which is any evaluation that feeds into a release decision.

Converting between formats

Moving data between formats is straightforward. The most common upgrade path is list → UC dataset as your project matures:

# List → UC dataset (most common upgrade) eval_data = [{"inputs": {"question": "What is Delta Lake?"}, "expectations": {"expected_facts": ["ACID", "open-source"]}}] eval_dataset = mlflow.genai.datasets.create_dataset(name="main.genai_eval.my_eval_v1") eval_dataset.merge_records(records=eval_data) # Spark → UC dataset spark_df = spark.table("main.analytics.curated_eval_cases") eval_dataset.merge_records(data=spark_df) # UC dataset → queryable via Spark spark.table("main.genai_eval.my_eval_v1").groupBy("tags.domain").count().display()
Decision framework

Two questions determine the right format: does this evaluation need to be reproducible? and does it need to be shared? If both are no — use a list. If reproducibility matters but not sharing — Spark on a versioned Delta table. If both matter — and they do for anything entering a quality gate or shared with another person — use the UC-backed EvaluationDataset.

Lab: Build Dataset from Production Traces

Convert real production traces into evaluation data — the recommended path for enterprise deployments. You'll generate sample traces by calling your agent, query them with mlflow.search_traces() filtered by status and timestamp, and add them to your UC dataset via merge_records(traces=...). The auto-extracted inputs and outputs become your production quality baseline. Lab content is in the companion notebook.

Step 1
Generate traces
Call your traced agent 10 times with representative queries to simulate production traffic.
Step 2
Search & filter
mlflow.search_traces(filter_string="attributes.status = 'OK'") — only successful traces become eval rows.
Step 3
Merge into dataset
eval_dataset.merge_records(traces=traces) — MLflow auto-extracts inputs/outputs from the trace structure.
Step 4
Inspect extraction
Verify the auto-extracted inputs and outputs match what you expect in the dataset UI.
