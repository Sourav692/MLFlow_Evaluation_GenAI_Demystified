8.1 · Theory

Evaluation Cost Management at Scale

When GenAI evaluation runs against real production traffic, costs explode silently. Every evaluation call is itself an LLM call — and naïve setups run every judge on every trace. This module is about keeping evaluation comprehensive but affordable.

Naïve evaluation runs every judge on every trace. The math compounds fast — your evaluation bill can exceed your inference bill within weeks.
The cost multiplication trap 5 judges × 10,000 traces/day = 50,000 LLM calls/day = ~1.5M calls/month = $$$ before you've shipped a feature

The fix is three layered strategies: tiering, sampling, and judge model selection. Used together, mature setups run evaluation at 5–10% of inference cost. Above 25% means you're over-judging.

Strategy 1 — Tiered evaluation

Run the cheapest scorers first, escalate to expensive ones only when needed. Code-based scorers (regex, latency checks, schema validation) are essentially free and deterministic — let them do the heavy lifting. Reserve frontier LLM judges for cases where semantic reasoning is genuinely required.

Interactive: tiered evaluation funnel Hover each tier

Strategy 2 — Smart sampling

Not every quality dimension needs the same coverage. Safety must run on every trace (zero tolerance for missed violations), but quality drift only needs a representative sample to surface trends. Sampling rates are configured directly in ScorerSamplingConfig(sample_rate=...) when you call .register().start().

Interactive: sampling fan-out Hover a stream

The three streams run independently in the background — each sampled trace gets feedback attached, visible in the MLflow UI.

Strategy 3 — Judge model selection

The judge LLM itself is a cost dial. Match the model to the job. A common pattern: same judge logic, two registered instances — one cheap-model variant for monitoring, one frontier-model variant for the CI gate.

High-volume monitoring
Cheap model · 24/7 prod
Use databricks-claude-haiku-4-5 for always-on sampled judges. Acceptable variance, low cost per call.
Pre-deployment gate
Frontier · CI only
Use databricks-claude-opus-4-7 at the quality gate. Runs once per release, cost is bounded, accuracy is highest.
Custom domain
Calibrate first
Sometimes a cheap model with a good rubric beats a frontier model with a vague one. Measure Pearson against humans before choosing.

Cost tracking on Databricks

Two visibility surfaces matter: the AI Gateway usage metrics (request count, blocked requests, cost per model, latency P95 — visible in the serving endpoint UI) and the MLflow assessment pane (when LiteLLM is configured as the judge backend, individual judge cost appears alongside each scored trace).

The cost-control checklist
01 Push every check possible into code-based scorers with @scorer — they're free and deterministic.
02 Use sampling rates that match risk: 100% for safety, ~30% for quality, ~10% for the full suite.
03 Pick the cheapest judge model that still passes calibration (Pearson ≥ 0.7 vs human ratings).
04 Route both inference and evaluation traffic through AI Gateway so cost is one queryable number.
05 Treat the eval bill as a first-class metric — review it weekly alongside quality scores.
Target ratio

A mature setup typically runs evaluation at 5–10% of inference cost. Anything above 25% means you're over-judging and should re-tier.

8.2 · Theory

Expert Anti-Patterns & How to Fix Them

Five mistakes that quietly poison evaluation results — even in mature pipelines. Each one feels reasonable when written, then produces misleading numbers for months until someone notices the gap between scores and reality.

AP·01
Self-evaluation bias
+
The bug
Using the same model family to both generate output and judge it (e.g. Sonnet generates, Sonnet judges).
Why it fails
A model rates its own writing style, length conventions, and reasoning patterns favorably. Scores inflate by roughly 0.3–0.5 points compared to a neutral judge — so your dashboard looks healthier than reality.
The fix
Pick a judge from a different vendor or model family. Then run alignment — Pearson ≥ 0.7 against human ratings before trusting it.
AP·02
One judge for everything
+
The bug
A single "Quality" judge whose rubric tries to cover correctness, safety, style, and latency all at once.
Why it fails
A score of 3.5 tells you nothing actionable. Did safety drop? Did formatting break? You can't write an alert on it, and you can't fix what you can't see.
The fix
Decompose into one scorer per orthogonal dimension. Use code scorers for deterministic axes (latency, format) and LLM judges only for semantic ones (correctness, safety, tone).
AP·03
Vague rubrics
+
The bug
"Score 1–5 based on quality."
Why it fails
With no anchors, judge scores become essentially random run-to-run. You'll see swings of ±1.0 between identical evaluation runs of the same data.
The fix
Anchor every level concretely. The good rubric is reproducible. The bad one is a coin flip dressed up as a metric.
Bad rubric
"Rate 1–5 based on quality"
Good rubric
5 = Precise, correct, includes caveats 4 = Mostly correct, minor omissions 3 = Partially correct, concept incomplete 2 = Significant inaccuracy 1 = Fundamentally wrong / misleading
AP·04
Static eval dataset
+
The bug
Same 50 examples, evaluated for six months, while production reveals new failure modes weekly.
Why it fails
Your benchmark goes stale. Real regressions never surface in pre-deployment evals — you ship them, then discover them from user complaints.
The fix
Treat the dataset as a living UC asset. Every interesting production failure becomes a new row. Practical rule: if the dataset hasn't grown in 30 days, it's probably stale.
AP·05
Skipping code scorers
+
The bug
Using LLM judges to check things a regex can answer — JSON validity, response length, citation presence.
Why it fails
Wastes evaluation budget and introduces variance. An LLM judge will sometimes say "yes, valid JSON" for malformed output. json.loads() won't.
The fix
Code scorers for anything deterministic; LLM judges only when semantic understanding is genuinely required.
Use a code scorer when… Use an LLM judge when…
Checking format or schema Checking factual correctness
Measuring latency or length Evaluating tone or relevance
Verifying required fields exist Detecting subtle hallucination
Comparing exact strings Assessing reasoning quality
Quick sanity checklist
? Could a regex check this? → use a code scorer, not an LLM
? Is the judge model the same family as the generator? → swap it
? Could one bad row score 3.5 and look fine? → decompose
? Has the dataset grown in the last 30 days? → it should
? Does each rubric level have a concrete description? → required

If any answer is "no", that's technical debt that will surface later as a quality regression you didn't see coming.

8.3 · Theory

The Complete Enterprise Eval Architecture

Every piece you've learned across Modules 1–7 fits into a single operating system. Mature Databricks GenAI evaluation runs as four layers in a closed loop: Dev builds, CI gates, Production monitors, and Feedback flows the truth back into Dev. Each layer hands off cleanly to the next, and the final layer feeds back to the first — that's what makes it a loop, not a one-way pipeline.

Reference: four-layer eval architecture End-to-end
LAYER · 1 Dev
Build dataset generate_eval_df + curated edge cases + prod traces. Saved as versioned UC asset.
evaluate() Run against local predict_fn. Mix code scorers (cheap) with LLM judges (semantic).
Experiment Compare runs side-by-side in MLflow UI. Iterate prompt, model, retrieval until bar is cleared.
Output: a model_id + a passing scorer suite, ready for promotion.
LAYER · 2 CI
UC dataset Same versioned asset from Layer 1 — governance and lineage carried through.
Endpoint eval to_predict_fn("endpoints:/...") hits deployed candidate, no code changes.
Quality gate Threshold logic. dbutils.notebook.exit("FAILED") blocks deploy task.
Effect: regressions can't ship — humans don't have to remember to check.
LAYER · 3 Production
AI Gateway Synchronous guardrails — PII, toxicity, topic limits — block bad I/O before the model.
Inference table Every request + response + full trace JSON lands in a UC Delta table — the audit trail.
Live monitors Registered scorers sample traces in background and attach feedback. Drift surfaces in UI.
Pattern: Gateway is fast-and-blocking, registered scorers are async-and-analytical — layered defense.
LAYER · 4 Feedback
User feedback Thumbs-up/down via log_feedback(), attached to trace by client_request_id.
Review App Domain experts label sampled traces. Labels stored as Feedback objects.
Alignment Calibrate automated judges vs human ratings. Pearson ≥ 0.7 before trusting them.
The layer most teams skip — and the reason most eval systems decay. Improvements flow back to Layer 1.

Why the loop matters

At every stage
Quality is measurable
Same scorer logic, three contexts — local eval, CI gate, production monitor.
Mechanically enforced
Regressions can't ship
Layer 2 blocks them automatically against the same dataset Layer 1 used.
Closed loop
Reality flows back
Layer 4 keeps the dataset and judges honest as user behavior shifts.
Governed
Lineage end-to-end
UC ties dataset, model, endpoint, inference table, eval runs into one auditable graph.
The maturity test

A team can adopt one layer at a time, but the compounding value comes from running all four. That's the difference between evaluating a GenAI system and operating one.

8.4 · Theory

LLMOps — Versioning, Lineage & Governance

In LLMOps, "ship the model" is too narrow. What you actually ship is a bundle — model + prompt + dataset + scorer suite — all versioned, lineage-tracked, and gated by a passing evaluation. This module is about treating evaluation as a first-class governance object, not a sidecar log.

Versioning: artifacts that move together

A model version on its own is meaningless. Without the prompt that drives it, the dataset it was scored against, and the scorers that approved it, you can't reproduce or audit anything. MLflow 3 ties these into a single bundle keyed by model_id — a stable identifier like models:/agent/3 that all related artifacts point to.

Versioned artifact bundle Bound by model_id

What's versioned, and where:

Models
Logged Models
Immutable artifact in MLflow, addressable as models:/agent/N.
Prompts
Prompt Registry
Versions and compares prompts alongside model versions.
Datasets
UC-backed Delta
Attached to experiment, with merge_records history for additions.
Scorers
Registered suites
Each scorer named uniquely (prod_safety_v1) and independently versioned.
Eval runs
Bound to model_id
Every evaluate() call passes model_id=..., linking scores to the bundle.

Lineage: the UC chain

Unity Catalog stitches the bundle into a queryable graph. From any node you can walk forward (what depends on this?) or backward (what produced this?). No manual logging — UC captures lineage automatically when you read or write a tracked asset.

Unity Catalog lineage chain Walk either direction

Audit questions that become trivial SQL queries against UC:

# Which model versions were evaluated against this dataset? SELECT model_id, run_id, metrics FROM uc.lineage.eval_runs WHERE dataset_name = 'tutorial_eval_v1'; # Which prod traces fed evaluation run exp_id=42? SELECT request_id, response, trace FROM uc.lineage.inference_tables WHERE evaluation_run_id = 42; # What scorer suite approved models:/agent/3 for promotion? SELECT scorer_name, version, threshold, last_score FROM uc.lineage.model_approvals WHERE model_id = 'models:/agent/3';

When a regulator, a compliance team, or a curious engineer asks these questions, the answer lives in the catalog — not in someone's notebook history.

Governance: evaluation as a checkpoint

The single most important rule of mature LLMOps:

The promotion rule
"No model version reaches production without a passing evaluation run attached to it."

This rule is enforced mechanically, not by convention. UC tags each model version with its last passing eval. The deployment task in the Workflow checks for that tag before pushing. If the eval is missing, stale, or below threshold, the deploy fails — no human needs to remember to check.

The cultural shift

Evaluation stops being "something we should do" and becomes the thing that lets us ship.

Deployment Jobs in MLflow 3

MLflow 3 introduces a managed deployment lifecycle — a single object that orchestrates four stages and logs each transition as a tagged event on the model version. Each stage is visible in the UC lineage graph, so "when did v.3 go live, who approved it, against which dataset?" is one query away.

Stage 1
Evaluate
Runs the registered scorer suite against the candidate model_id.
Stage 2
Approve
Auto-approve on clean thresholds, or route to human reviewer.
Stage 3
Deploy
Pushes to Databricks Model Serving with AI Gateway pre-configured.
Stage 4
Monitor
Registered scorers begin sampling live traffic immediately.
What to demand from any LLMOps stack
✓ Every artifact is versioned: model, prompt, dataset, scorer suite
✓ Every evaluation run links to a model_id
✓ Lineage is queryable end-to-end via Unity Catalog
✓ No promotion path bypasses the evaluation gate
✓ Deployment events are tagged on the model version, not in a sidecar log
If any of these is missing, you don't have LLMOps — you have manual ops with extra steps.
