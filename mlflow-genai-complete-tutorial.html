8.1 Evaluation Cost Management at Scale
Theory
The cost multiplication trap: 5 judges × 10,000 prod traces/day = 50,000 LLM calls for evaluation alone
Tiered evaluation: cheap code scorers first → expensive LLM judges only on failures
Smart sampling: 100% safety + 30% quality + 10% full judge suite
Judge model selection for cost: use smaller models for high-volume monitoring, frontier models for pre-deployment gate
Databricks cost tracking: AI Gateway usage metrics + MLflow judge cost in assessment pane (requires LiteLLM)
8.2 Expert Anti-Patterns & How to Fix Them
Theory
❌ Self-evaluation bias: judging GPT-4o output with GPT-4o — use a different judge model
❌ One judge for everything: single Correctness judge missing safety, style, latency — decompose
❌ Vague rubrics: "score 1–5 based on quality" → judge scores are random — add concrete level descriptions
❌ Static eval dataset: not adding new failure modes as they're discovered — treat dataset as living artifact
❌ Skipping code scorers: using LLM judges for things regex can check — wastes budget and introduces variance
8.3 LLMOps — Versioning, Lineage & Governance on Databricks
Theory
Logged Models + model_id in evaluate(): links every evaluation run to a specific model version
Deployment Jobs in MLflow 3: managed lifecycle — evaluate → approve → deploy → monitor
Unity Catalog lineage: dataset → model → endpoint → inference table → evaluation — fully auditable
Prompt Registry: version and compare prompts alongside model versions in MLflow
Evaluation as a governance checkpoint: no model version moves to production without a passing eval run attached
