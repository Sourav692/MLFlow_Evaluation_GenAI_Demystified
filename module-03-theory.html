3.1 · Theory

How predict_fn Works Inside mlflow.genai.evaluate()

The predict_fn is the bridge between your AI agent and MLflow's evaluation harness. It tells evaluate() how to call your agent for each row in your dataset. Understanding this mechanism is the prerequisite for every pattern in this module.

Three evaluation modes

mlflow.genai.evaluate() operates in three distinct modes depending on what you provide. The mode determines whether MLflow calls your agent live, hits a remote endpoint, or simply scores pre-existing outputs.

Mode 1
Pre-generated outputs
You already have agent responses in your dataset. No predict_fn needed — MLflow just scores the existing outputs. Useful for evaluating historical data or comparing cached responses.
Mode 2
Live predict_fn
You pass a callable function. MLflow calls it once per dataset row, collects the output and trace, then scores everything. The primary pattern for development-time evaluation.
Mode 3
Deployed endpoint via to_predict_fn()
You wrap a serving endpoint with to_predict_fn(). MLflow calls the endpoint remotely, retrieves the trace, and scores it. The pattern for CI gates and production testing.
Interactive — Per-row evaluation pipeline

Step 1 — Pick a row from the dataset

MLflow extracts the inputs dict, expectations (if present), and any tags for this row.

# What the row looks like internally: row = { "inputs": {"question": "What is Z-ordering in Delta?"}, "expectations": {"expected_facts": ["data skipping", "multi-dimensional clustering"]} }

Step 2 — Unpack inputs as keyword arguments

MLflow calls predict_fn(**row["inputs"]). This is why parameter names must match exactly — a key named question in the dataset must correspond to a parameter named question in your function.

# MLflow does this internally: output = predict_fn(**{"question": "What is Z-ordering in Delta?"}) # ...which means: output = predict_fn(question="What is Z-ordering in Delta?")

Step 3 — Call predict_fn, capture trace

Your function runs, calls the LLM, returns a response. MLflow captures the trace automatically via @mlflow.trace or autolog(). One trace is linked to this evaluation row.

@mlflow.trace # ← this is what captures the trace def my_agent(question: str) -> str: resp = client.chat.completions.create(...) return resp.choices[0].message.content # ← output captured

Step 4 — Pass everything to scorers in parallel

Each scorer receives: inputs, outputs (your return value), expectations, and the full trace. All scorers for this row run concurrently in a thread pool.

# Each scorer gets this signature: def scorer( inputs: dict, # {"question": "..."} outputs: str, # your predict_fn return value expectations: dict, # {"expected_facts": [...]} trace: Trace # full MLflow trace with spans ) -> Feedback

Step 5 — Collect Feedback, attach to trace

Each scorer returns a Feedback object with value, rationale, and source. These appear as assessments on the trace in the MLflow UI. After all rows complete, MLflow aggregates per-row scores into run-level metrics.

# Run-level metrics after all rows: { "correctness/mean": 4.2, # average across rows "safety/ratio_pass": 0.97, # fraction passing "relevance_to_query/mean": 3.8, # average across rows }
Step 1 of 5

The predict_fn contract

When you pass a predict_fn to evaluate(), MLflow expects it to follow a strict contract. Violating any of these rules produces confusing errors or silently broken evaluations.

Rule 1
Parameter names must match inputs keys exactly
If your dataset has inputs: {"question": "..."}, your function must be def my_fn(question: str). MLflow unpacks the inputs dict as keyword arguments.
Most common error
Rule 2
Each call must emit exactly one MLflow trace
Use @mlflow.trace or enable mlflow.openai.autolog(). The trace is how MLflow links the row to the agent's internal spans. Without it, trace-based scorers silently fail.
@mlflow.trace
Rule 3
Return value must be JSON-serializable
A plain string for Q&A agents, or a dict for structured outputs. Lists and nested dicts are fine. Custom objects or generator outputs will fail serialization.
str / dict
Key insight — concurrency

MLflow runs predict_fn and scorers concurrently via thread pools. Multiple rows are processed in parallel, and all scorers for a given row also run in parallel. Two env vars control this: MLFLOW_GENAI_EVAL_MAX_WORKERS for row parallelism and MLFLOW_GENAI_EVAL_MAX_SCORER_WORKERS for scorer parallelism.

3.2 · Lab

Pattern 1 — Plain Python Function (Local Agent)

The most common pattern during development. You define your agent as a regular Python function in your notebook, decorate it for tracing, and pass it directly to evaluate(). No deployment needed — iterate on prompts and logic in real time.

from databricks_openai import DatabricksOpenAI client = DatabricksOpenAI() @mlflow.trace def my_agent(question: str) -> str: resp = client.chat.completions.create( model="databricks-claude-sonnet-4", messages=[ {"role": "system", "content": "You are a Databricks expert."}, {"role": "user", "content": question} ] ) return resp.choices[0].message.content results = mlflow.genai.evaluate( data=eval_dataset, predict_fn=my_agent, scorers=[Correctness(), RelevanceToQuery()], model_id="models:/my-agent/1" # lineage tag, not execution )

The critical detail: question in the function signature must match the key name in the dataset's inputs dict. If the dataset has {"inputs": {"question": "..."}}, the function parameter must be named question — not q, not query, not prompt.

If you use mlflow.openai.autolog(), traces are captured automatically for OpenAI-compatible clients — you don't need the @mlflow.trace decorator. But one of the two is required.

Common pitfalls

Pattern 1 — failure modes and fixes
Pitfall
Parameter name mismatch — def my_agent(q: str) with dataset inputs: {"question": "..."}
TypeError
Fix
Rename parameter to question — or rename the dataset key to q. They must be identical.
Pitfall
Heavy init inside predict_fn — creating DatabricksOpenAI() inside the function. Called N times.
Slow
Fix
Move client creation outside the function. Initialize once, reference many times.
Pitfall
No trace emitted — forgot @mlflow.trace and autolog is off. Trace-based scorers silently produce null.
Silent fail
Fix
Always add @mlflow.trace or call mlflow.openai.autolog() before defining the function.
model_id is metadata, not execution

The model_id parameter in evaluate() doesn't affect how your agent runs — it's a lineage tag that links the evaluation run to a specific model version in MLflow. This becomes essential when you set up CI quality gates in Module 7.

3.3 · Lab

Pattern 2 — to_predict_fn() for Deployed Endpoints

When your agent is already deployed to Databricks Model Serving, you don't rewrite your agent as a local function. to_predict_fn() wraps the endpoint into something evaluate() can call — translating inputs to the endpoint's payload format, adding return_trace=True, and copying the trace back to your experiment.

to_predict_fn() — remote endpoint evaluation architecture
import mlflow from mlflow.genai.scorers import RelevanceToQuery, Correctness # Step 1: Wrap the deployed endpoint predict_fn = mlflow.genai.to_predict_fn("endpoints:/my-databricks-agent") # Step 2: Data MUST match endpoint schema (chat = messages key) endpoint_eval_data = [ {"inputs": {"messages": [{"role": "user", "content": "What is Z-ordering?"}]}, "expectations": {"expected_facts": ["data skipping", "multi-dimensional clustering"]}}, ] # Step 3: Same evaluate() call — pattern is transparent results = mlflow.genai.evaluate( data=endpoint_eval_data, predict_fn=predict_fn, scorers=[Correctness(), RelevanceToQuery()] )

The input format catch

This is the detail that trips people up most often. The inputs format in your dataset must match the endpoint's expected schema, not the simpler format you'd use with a local function.

Pattern Dataset inputs format Why
Pattern 1 (local) {"question": "What is Z-ordering?"} Matches your function signature directly
Pattern 2 (endpoint) {"messages": [{"role": "user", ...}]} Must match the endpoint's HTTP payload schema
Sending {"question": "..."} to a chat endpoint produces a schema mismatch error. The wrapper translates your inputs into the HTTP payload, but it can't guess the format — you have to provide it correctly.
Required
Tracing enabled
Either deployed via mlflow.deploy() (auto-enables) or set ENABLE_MLFLOW_TRACING=True in endpoint env vars.
Required
Endpoint is live
The wrapper makes real HTTP calls. Endpoint must be running and accessible from your notebook.
Required
Correct input schema
Know whether the endpoint expects messages, question, or another key structure.
Required
Experiment set
Traces are copied to the current experiment. Call mlflow.set_experiment() before evaluate().
When to use Pattern 2

Use to_predict_fn() when you need to validate the deployed version — the serving environment (dependencies, compute, configuration) can cause behaviour differences from local code. This is the standard pattern for CI quality gates and A/B endpoint comparisons.

3.4 · Theory

Pattern 3 — Async predict_fn & Registered Models

These two patterns solve different problems but share a common theme: optimising how your agent gets called during evaluation. Async functions maximise throughput by overlapping network waits. Registered models avoid redundant loading.

Interactive — execution timeline comparison
Total wall-clock time: ~12.0s
from openai import AsyncOpenAI async_client = AsyncOpenAI(...) async def my_async_agent(question: str) -> str: resp = await async_client.chat.completions.create( model="databricks-claude-sonnet-4", messages=[{"role": "user", "content": question}] ) return resp.choices[0].message.content # MLflow auto-detects async — just pass it in results = mlflow.genai.evaluate( data=eval_dataset, predict_fn=my_async_agent, scorers=[Correctness()] )

Two environment variables control async behaviour: MLFLOW_GENAI_EVAL_ASYNC_TIMEOUT (default 5 minutes per call — increase for complex multi-step agents) and MLFLOW_GENAI_EVAL_MAX_WORKERS (controls concurrency — start at 4 for Databricks Foundation Model APIs).

Registered model pattern

This pattern solves a different problem: avoiding redundant model loading. If you call mlflow.pyfunc.load_model() inside your predict_fn, the model gets loaded from scratch on every single evaluation row.

Approach Code pattern Cost for 50 rows Verdict
Anti-pattern load_model() inside predict_fn 50 × 3s load = 2.5 min wasted Avoid
Correct load_model() once, outside 3s load once, then near-zero Use this
# Load the model ONCE, outside predict_fn model = mlflow.pyfunc.load_model("models:/my-agent/3") # predict_fn just calls .predict() — no loading overhead def predict_fn(question: str) -> str: return model.predict({"question": question}) results = mlflow.genai.evaluate( data=eval_dataset, predict_fn=predict_fn, scorers=[Correctness()], model_id="models:/my-agent/3" )

Rate limiting & backoff

When running evaluations against Databricks Foundation Model APIs with larger datasets, you will hit rate limits. The recommended approach is to wrap your predict_fn with retry logic using exponential backoff. This combines cleanly with both sync and async patterns.

from tenacity import retry, wait_exponential, stop_after_attempt @retry(wait=wait_exponential(min=1, max=30), stop=stop_after_attempt(5)) @mlflow.trace def my_agent(question: str) -> str: resp = client.chat.completions.create( model="databricks-claude-sonnet-4", messages=[{"role": "user", "content": question}] ) return resp.choices[0].message.content
For Databricks serverless environments, start with MLFLOW_GENAI_EVAL_MAX_WORKERS=4 and tune up based on endpoint throughput. Pushing too high causes 429 errors and actually slows things down due to retries.
3.5 · Theory

Pattern 4 — Multi-Turn with ConversationSimulator

All previous patterns evaluate single-turn interactions: one question in, one answer out. But many real agents handle multi-turn conversations — customer support, onboarding wizards, research assistants — where quality depends on how well the agent maintains context across turns. ConversationSimulator addresses exactly this.

Failure mode 1
Context amnesia
Agent gives a great recommendation in turn 1, then in turn 3 suggests something contradictory — as if the earlier exchange never happened.
Failure mode 2
Repetitive responses
Agent gives the same generic advice in turns 2 and 4 instead of building on what was already discussed.
Failure mode 3
Goal drift
User wanted pipeline setup help, but the agent keeps answering tangential questions without steering back.
Failure mode 4
Escalation failure
User signals frustration ("I already tried that"), but the agent doesn't adjust its tone or approach.
Interactive — ConversationSimulator replay PERSONA: Frustrated admin
U
I've been trying to grant my team access to a Unity Catalog schema for hours. The docs say to use GRANT but it keeps failing with a permissions error.
A
To grant access to a Unity Catalog schema, you'll need to be the schema owner or have MANAGE permissions. Try: GRANT USE SCHEMA ON SCHEMA catalog.schema TO `group_name`
U
I already tried that exact command. It says "User does not have MANAGE on schema." I'm a workspace admin — shouldn't I have all permissions?
A
Good catch — workspace admin and Unity Catalog admin are separate roles. Being a workspace admin doesn't automatically grant UC permissions. You need the metastore admin to grant you ownership or MANAGE on that schema first.
U
OK, who is the metastore admin? How do I find out?
A
In your Databricks workspace, go to the Admin Console → Unity Catalog tab. The metastore admin is listed at the top. If you're using an account-level metastore, you may need to check the account console instead.
KnowledgeRetention 5 / 5 — remembered user is workspace admin
TaskCompletion 3 / 5 — didn't resolve the grant, gave next step
UserFrustration Low — adapted to frustration cues well
@mlflow.trace def my_multi_turn_agent(input: list[dict]) -> str: """ input looks like: [ {"role": "user", "content": "Help me set up a Delta table"}, {"role": "assistant", "content": "Sure! First, let's..."}, {"role": "user", "content": "What about partitioning?"} ] """ resp = client.chat.completions.create( model="databricks-claude-sonnet-4", messages=[ {"role": "system", "content": "You are a Databricks expert."}, *input # pass full history ] ) return resp.choices[0].message.content

The critical signature change: instead of question: str, multi-turn agents accept input: list[dict] — the full conversation history in OpenAI Responses format. The simulator manages state, calling your function repeatedly and appending each exchange.

Multi-turn judges

Judge What it measures Failure it catches
KnowledgeRetention Does the agent remember facts from earlier turns? User said "I'm using Unity Catalog" in turn 1, agent asks "which catalog?" in turn 4
TaskCompletion Did the conversation achieve its stated goal? User wanted pipeline setup — conversation ended without completing it
UserFrustration Does the user need to repeat or rephrase? User says "I already tried that" but agent doesn't adjust its approach

Conversation scenario design

multi_turn_scenarios = [ { "inputs": { "user_persona": "A data engineer new to Databricks", "goal": "Set up a Delta Live Tables pipeline with quality constraints", "num_turns": 5 } }, { "inputs": { "user_persona": "A frustrated admin who already tried the docs", "goal": "Debug a failing Unity Catalog permission for a team", "num_turns": 4 } }, ]
Cost awareness

Multi-turn evaluation is significantly more expensive than single-turn. Each scenario requires multiple LLM calls per turn (one for the simulator, one for your agent), plus the judges at the end. Use it strategically for conversational agents and multi-step workflows — not for simple Q&A where each interaction is independent.

Pattern predict_fn signature Best for Tradeoff
1 — Local function def fn(question: str) -> str Development iteration — fast prompt and logic changes Tests local code, not deployed environment
