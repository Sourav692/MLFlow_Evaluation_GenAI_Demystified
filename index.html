WHY THIS COURSE EXISTS

LLM evaluation is not classical testing.

Every assumption that makes software testing work — determinism, fixed expected outputs, binary pass/fail — breaks the moment you put a language model in the loop. This is what changes.

Dimension Classical software testing LLM & agent evaluation
Input → output Deterministic — same input, same output Stochastic — same input, different outputs on every call
Correctness Binary pass/fail against fixed expected value Spectrum + judgment — multiple acceptable answers
Failure modes Logic bugs, exceptions, regressions Hallucinations, drift, safety violations, format errors
Metrics Coverage %, assertions passed Distribution of judge scores across thousands of traces
Test strategy Edge-case enumeration Statistical sampling + judge alignment + human review
Tooling pytest, JUnit, Jest mlflow.genai.evaluate(), MLflow Tracing, Inference Tables
Cost per run Negligible — milliseconds, no API spend $0.001 – $0.10 per LLM call · cost-aware design required
Debugging Stack trace tells you the line Trace + judge rationale + human review

CORE OBJECTS

Five primitives the whole stack rests on.

Everything in MLflow 3 GenAI evaluation reduces to these five objects. Knowing them cold makes every module easier — and makes the official docs readable in one pass.

Experiment

Top-level container grouping every evaluation run for a single project or model.

Run

One invocation of evaluate() with a dataset version, predict_fn, and scorers.

M01 →
Trace

Hierarchical execution log of one agent call — spans, inputs, outputs, latencies, token counts.

M05 →
Scorer

A function (code or LLM) that scores a single trace on one dimension — correctness, safety, latency, etc.

M04 →
Judge

A scorer backed by an LLM — built-in (Correctness) or custom via make_judge().

FREQUENTLY ASKED

Questions, before you commit a weekend.

The honest answers — prerequisites, what you need to run the labs, and how this differs from a generic LLM tutorial.

01What do I need before I start?
Working Python, comfort with notebooks, and basic familiarity with LLM APIs. You don't need an ML background — but you should be the kind of engineer who reads docs and writes pipelines. The first lab walks you through workspace setup end to end.
02Do I need a paid Databricks account?
You need a Databricks workspace with Unity Catalog and Model Serving enabled. Community Edition won't cover Modules 5–8 (Vector Search, AI Gateway, inference tables). Use a 14-day free trial or a sandbox workspace from your team.
03Which clouds does this work on?
Anywhere Databricks runs — AWS, Azure, or GCP. The labs use only Databricks-native primitives (UC, Vector Search, Model Serving, AI Gateway), so the same notebooks work across all three. Where cloud differs, the lab notes it.
04How is this different from a generic LLM tutorial?
Most tutorials stop at "call evaluate() on a toy dataset." This course covers the full production loop — UC-governed datasets, judge alignment, AI Gateway guardrails, inference tables, registered async monitoring, and CI quality gates. You finish with a pipeline you can actually defend in a design review.
05How long does it really take?
About ~24 hours end to end if you do every lab. The capstone alone (Module 8) is ~2.5 hours. Most people spread it across two to three weekends; teams have run it as a 3-day internal workshop.
06Is there a cost or certificate?
The course is free and the source is open. There's no formal certificate — your portfolio is the capstone pipeline you build (registered models, deployed endpoints, traced evals in your own UC). You'll pay only your Databricks compute for running the labs.
